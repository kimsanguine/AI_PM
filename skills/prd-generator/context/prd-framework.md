## PRD 품질 기준

좋은 PRD는 다음 5가지 질문에 명확히 답합니다:

1. **Why** — 왜 이 문제를 지금 해결해야 하는가?
2. **What** — 정확히 무엇을 만드는가? (스코프의 경계)
3. **Who** — 누구를 위한 것인가? (타겟 세그먼트)
4. **How Much** — 성공을 어떻게 측정하는가? (지표)
5. **What If Not** — 안 하면 어떻게 되는가? (기회비용)

## PRD 유형별 깊이 가이드

### 신규 기능 (New Feature)
- 문제 정의 깊이: 높음 (사용자 문제 + 비즈니스 문제 모두)
- 솔루션 옵션: 3~5개 필수
- 기술 고려사항: 중간 (아키텍처 영향 중심)
- 타임라인: 마일스톤 단위

### 기능 개선 (Enhancement)
- 문제 정의 깊이: 중간 (현재 지표 vs 목표 지표)
- 솔루션 옵션: 2~3개
- 기술 고려사항: 낮음 (기존 아키텍처 내)
- 타임라인: 스프린트 단위

### 실험 (Experiment)
- 구조 변경: 문제 → **가설** → 실험 설계 → 성공 기준
- 솔루션 옵션: 실험 변수(variant) 설계로 대체
- 필수 포함: 최소 샘플 사이즈, 실험 기간, 통계적 유의성 기준

### 기술 부채 (Tech Debt)
- 문제 정의: 현재 기술 부채가 주는 비용 (개발 속도, 장애 빈도, 온보딩 비용)
- 솔루션: Before/After 아키텍처 비교
- 필수 포함: 안 하면 6개월/1년 후 예상 비용

## 우선순위 프레임워크

### RICE 스코어링
```
RICE = (Reach × Impact × Confidence) / Effort

Reach: 분기 내 영향받는 사용자 수
Impact: 0.25 (최소) / 0.5 (낮음) / 1 (보통) / 2 (높음) / 3 (매우 높음)
Confidence: 100% (높음) / 80% (중간) / 50% (낮음)
Effort: person-weeks
```

### 우선순위 매핑
- **P0 (Must)**: RICE 상위 10%, 또는 매출/사용자 안전에 직접 영향
- **P1 (Should)**: RICE 상위 30%, 분기 목표에 직접 기여
- **P2 (Could)**: 나머지, 리소스 여유 시 진행

## 지표 설정 가이드

### OMTM (One Metric That Matters)
- PRD당 하나의 핵심 지표만 선택
- "이 기능이 성공했다"를 한 문장으로 말할 수 있어야 함

### 보조 지표
- OMTM을 보완하는 2~3개 지표
- "왜 OMTM이 움직였는지"를 설명하는 선행/후행 지표

### 가드레일 지표
- "이 지표가 악화되면 즉시 롤백"하는 방어 지표
- 예: 앱 크래시율, 기존 기능 전환율, 고객 문의 급증

## 이해관계자별 PRD 읽기 패턴

| 독자 | 먼저 보는 섹션 | 깊게 보는 섹션 | 무시하는 섹션 |
| --- | --- | --- | --- |
| CEO/경영진 | Executive Summary | 목표 & 지표, 타임라인 | 기술 고려사항 |
| 엔지니어링 리드 | 솔루션 개요 | 기술 고려사항, 스코프 | Executive Summary |
| 디자이너 | 사용자 스토리 | 플로우, 엣지 케이스 | 기술 고려사항 |
| QA | 스코프 | 엣지 케이스, 성능 요구사항 | 문제 정의 |

→ PRD의 각 섹션은 "누가 읽는가"를 의식하며 작성합니다.
