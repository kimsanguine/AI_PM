# /pm-setup — 대화형 PM 컨텍스트 부트스트랩

## Task

비개발자 PM 이 새 프로젝트에서 Claude Code 를 처음 쓸 때, **제품·타깃·메트릭을 소크라틱 인터뷰로 물어 `CLAUDE.md` 8축 초안 + `INDEX.md` 지식 지도 초안을 자동 작성** 한다. 한 번의 대화로 "AI 가 우리 제품을 안다" 상태를 만드는 것이 목적이다.

## Scope

- 인터뷰 + 두 초안 파일 작성까지. 실제 코드 작성·기능 구현·외부 도구 연결은 하지 않는다.
- 입력: 사용자 답변. 답변이 없거나 모르면 추측하지 않고 `[확인 필요]` 로 비워 둔다.
- 출력 파일명: `CLAUDE.draft.md`, `INDEX.draft.md` 로 저장한다. 기존 `CLAUDE.md`·`INDEX.md` 가 있어도 **덮어쓰지 않는다** — 사용자가 검토 후 직접 채택한다.
- 한 번에 한 묶음만 묻는다. 8개를 한꺼번에 쏟지 않는다(인터뷰 피로 방지).

## Length

- 인터뷰 질문은 묶음당 2–4개. `CLAUDE.draft.md` 본문 1,500–3,000자, `INDEX.draft.md` 800–2,000자.
- 사용자 답변이 짧아도 초안은 8축 골격을 모두 갖춘다(빈 축은 `[확인 필요]`).

## Format

### 1단계 — 소크라틱 인터뷰 (묶음별로 순차 진행)

기존 소크라틱 PRD 방법론(`templates/commands/prd.md`)을 재사용한다. 아래 5묶음을 **순서대로** 묻고, 각 묶음 답을 받은 뒤 다음으로 넘어간다.

- **A. 제품**: 이 제품은 누구의 어떤 문제를 해결하나? 한 문장으로? 자매/경쟁 제품은?
- **B. 타깃**: 핵심 사용자 세그먼트는? 그들이 지금 쓰는 대안은? 우리가 안 만드는 사용자는?
- **C. 메트릭**: 성공을 하나의 숫자로 본다면(OMTM)? 악화되면 안 되는 가드레일 지표는?
- **D. 글쓰기/톤**: 산출물 언어(한국어/영어 혼용 규칙)? 금지 표현·금지 형용사가 있나?
- **E. 지식 위치**: 자료는 지금 어디에 흩어져 있나(폴더·문서·외부 도구)? 무엇을 SoT 로 둘까?

각 답이 모호하면 추측하지 말고 한 단계 더 캐묻는다("그 '빠르다' 는 무엇 대비 얼마나?"). 사용자가 "모름" 이라 하면 그 축은 `[확인 필요]` 로 둔다.

### 2단계 — `CLAUDE.draft.md` (8축 골격)

```markdown
# CLAUDE.md — <제품명> (초안)

## 1. Project — 정체성
<A 답변: 제품 한 문장 · 자매/경쟁>

## 2. Conventions — 글쓰기 규칙
<D 답변: 언어 · 구조 · 금지 표현>

## 3. Workflows — 작업 규칙
<반복 작업 · 슬래시/스킬 4섹션 의무 여부>

## 4. Tools — 도구·모델
- 모델: claude-opus-4-8
<연결할 MCP/도구 · effort 정책>

## 5. Voice — 톤
<직설/완곡 · 동의 기준>

## 6. Examples — 예시
<good/bad 짝 제공 원칙>

## 7. Escalation / Out-of-scope
<일반화 금지 · 추측 시 [확인 필요] 라벨>

## 8. Knowledge Map — 지식 진입점
<INDEX.draft.md 를 먼저 읽는다 · 변경 시 갱신>

---
last_updated: <오늘 날짜>
claude_model: claude-opus-4-8
```

### 3단계 — `INDEX.draft.md` (지식 지도)

```markdown
# INDEX.md — 지식 지도 (초안)

> 작업 전 이 파일을 먼저 읽고 관련 파일만 펼친다.
> 형식: `경로` · owner · use · freshness

## 운영
- `CLAUDE.md` · owner: <이름> · use: 행동 규칙(8축) · freshness: <YYYY-MM>
- `INDEX.md` · owner: <이름> · use: 지식 지도 · freshness: <YYYY-MM>

## <E 답변에서 도출한 폴더/영역>
- `<경로>` · use: <설명>

## 외부 (참조)
- <E 답변: SoT 도구 · 외부 자료 위치>
```

### 4단계 — 마무리 보고

- 채워진 축 / `[확인 필요]` 로 남은 축을 한 줄 요약으로 보고한다.
- 사용자에게 "검토 후 `CLAUDE.md`·`INDEX.md` 로 채택할지" 를 묻는다(자동 채택 금지).

## 룰

- 출처 명시: 8축 CLAUDE.md 골격과 INDEX 지식 지도 형식은 본 레포 `CLAUDE.md`·`INDEX.md` 구조를 따른다. 소크라틱 인터뷰는 `templates/commands/prd.md` 방법론을 재사용한다.
- 추측 금지: 사용자가 말하지 않은 사실(메트릭 수치·경쟁사·사용자 수)을 채우지 않는다. 비우고 `[확인 필요]`.
- 한 번에 한 묶음. 5묶음을 한꺼번에 묻지 않는다.
- 마케팅 형용사("혁신적", "최고") 금지 — 제품 정의는 구체적 문장으로.
- 기존 파일 덮어쓰기 금지. 출력은 항상 `*.draft.md`.

## Examples

<example>
<good>
A 묶음 답변: "중소 제조사 구매 담당자가 자재 발주 시점을 놓치는 문제를 푼다. 한 문장: '재고가 임계치에 닿으면 자동으로 발주 초안을 만든다.'"
→ ## 1. Project 에 그대로 반영. 경쟁사를 안 물었으면 "자매/경쟁: [확인 필요]".
</good>
<bad>
A 묶음 답변이 "그냥 좋은 발주 도구" 인데, 초안에 "업계 최고의 혁신적 발주 솔루션으로 효율을 3배 높인다" 라고 추측해서 채움.
</bad>
</example>

<example>
<good>
C 묶음에서 사용자가 "성공 지표 모르겠다" 고 답 → `## 2. 목표` 의 OMTM 을 `[확인 필요 — OMTM 미정]` 으로 비우고, 마무리 보고에서 "메트릭 축 미완 — 확정 후 채워야 함" 이라고 알림.
</good>
<bad>
사용자가 모른다는데도 "OMTM 은 MAU 입니다" 라고 임의로 단정.
</bad>
</example>

<example>
<good>
5묶음을 A→B→C→D→E 순서로 하나씩 묻고, 각 답을 받은 뒤 다음으로 진행.
</good>
<bad>
첫 메시지에서 8축 질문 20개를 한꺼번에 쏟아 사용자가 어디서부터 답할지 모르게 만듦.
</bad>
</example>
