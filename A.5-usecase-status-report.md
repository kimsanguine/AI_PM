# 유즈케이스: 프로젝트 상태 보고서 자동 생성

> **PM 역량**: 이해관계자 커뮤니케이션 / 프로젝트 가시성 / 리스크 관리
> **난이도**: J → P
> **관련 문서**: [3.1-mcp-integration.md](./3.1-mcp-integration.md), [7.2-growth-kpi-dashboard.md](./7.2-growth-kpi-dashboard.md)

---

## 왜 PM에게 필요한가?

주간 상태 보고서는 PM의 핵심 산출물이지만, 가장 반복적인 업무이기도 합니다. Linear에서 이슈 수치를 뽑고, Notion에서 마일스톤을 확인하고, Slack에서 핵심 논의를 요약하고, 이 모든 것을 보고서 형식에 맞춰 정리하는 데 매주 1~2시간이 소요됩니다.

Claude Code는 이 과정을 **데이터 수집 → 분석 → 보고서 작성**으로 자동화합니다. PM은 "자동 생성된 초안을 리뷰하고 맥락을 추가"하는 역할로 전환됩니다.

---

## 상황

당신은 AI 더빙 서비스의 PM입니다. 매주 금요일 오후, VP of Product에게 프로젝트 상태 보고서를 제출해야 합니다. 보고서에는 스프린트 진행률, 핵심 지표 변화, 리스크, 다음 주 계획이 포함됩니다.

---

## 기존 방식 vs Claude 방식

```
기존 (1~2시간, 매주 반복):
1. Linear에서 이번 스프린트 통계 스크린샷 (10분)
2. 이슈별 진행 상황 수동 정리 (20분)
3. Notion에서 마일스톤 달성률 확인 (10분)
4. Slack에서 주요 논의 요약 (15분)
5. 지표 대시보드에서 수치 수집 (10분)
6. Google Slides/Notion 보고서 형식에 맞춰 작성 (20분)
7. 교정 및 맥락 추가 (15분)

Claude (15분):
1. 명령 입력 → 자동 수집 + 초안 생성 (3분)
2. PM 리뷰: 맥락 추가 + 판단 코멘트 (10분)
3. 최종 발행 (2분)
```

## 워크스루

### Step 1: 보고서 프레임워크 설정

```
Terminal 입력:
> 이번 주 프로젝트 상태 보고서를 작성해줘.

  ## 보고서 구조
  1. Executive Summary (3줄 이내)
  2. 스프린트 현황 (이번 스프린트 진행률, 번다운 상태)
  3. 핵심 지표 (전주 대비 변화)
  4. 완료된 항목 (이번 주 Done으로 전환된 이슈)
  5. 리스크 & 블로커 (현재 Blocked 이슈 + 잠재 리스크)
  6. 다음 주 계획 (예정된 릴리즈, 주요 작업)

  ## 데이터 소스
  - Linear: product-avatar 팀의 현재 스프린트 데이터
  - Slack: #product-avatar에서 이번 주 주요 논의 (스레드 5개 이상)
  - Notion: Q1 마일스톤 페이지의 진행률

  ## 형식 규칙
  - 각 항목에 [수치] 또는 [근거]를 반드시 포함
  - 리스크는 영향도(H/M/L)와 발생 확률(H/M/L) 매트릭스로 표시
  - 전주 보고서 대비 변화를 ↑↓→ 화살표로 표시

Claude Code 응답:
데이터를 수집하고 있습니다...

✅ Linear: Sprint W8 데이터 — 24개 이슈 로드
✅ Slack: 이번 주 주요 스레드 8개 수집

## PM 판단 포인트

### ❓ "자동 생성된 보고서를 그대로 보내도 되나?"

**절대 안 됩니다.** 자동 생성 보고서는 "초안"입니다. PM이 추가해야 하는 것:

```
Claude가 잘하는 것:
- 수치 집계 (이슈 수, 완료율, 지표 변화)
- 형식 일관성 유지
- 전주 대비 비교
- 데이터 기반 리스크 플래깅

PM이 반드시 추가해야 하는 것:
1. [내러티브] "왜 완료율이 하락했는가"의 실제 맥락
   → "블로커 2건은 외부 벤더 이슈로, 내부 역량 문제 아님"
2. [정치적 맥락] VP가 특별히 관심 갖는 항목 강조
   → "Enterprise 데모 리스크를 최상단에 배치"
3. [전략적 프레이밍] 수치 뒤의 의미 해석
   → "에러율 소폭 상승이지만 렌더링 성능 52% 개선이 더 큰 임팩트"
4. [결정 요청] 보고서에 포함할 의사결정 요청사항
   → "온보딩 A/B 결과 기반 Go/No-Go 결정 필요"
```

### ❓ "보고서의 숫자가 정확한지 어떻게 확인하나?"

```
검증 전략:
1. [샘플 체크] Done 이슈 3개를 Linear에서 직접 열어 상태 확인
2. [총합 검증] 전체 이슈 수 24 = Done 18 + IP 3 + Blocked 2 + BL 1
3. [지표 교차 검증] DAU 수치를 대시보드에서 직접 확인
4. [트렌드 상식 확인] 전주 대비 변화가 상식적 범위인지
   → 예: DAU가 갑자기 50% 급등했다면 데이터 오류 의심
```

반복 사용을 위해 SKILL.md로 저장합니다.

```

  SKILL.md를 생성해줘.

  이 스킬은:
  - 매주 금요일 오후 3시에 실행

  - 표준 보고서 형식으로 초안 생성
  - /docs/status-reports/YYYY-WXX.md로 저장

## 실습 과제

### Level 1 (J): 첫 상태 보고서
자신의 Linear (또는 Jira) 데이터로 주간 보고서를 생성하세요. 자동 생성된 수치 3개를 직접 검증하세요.

### Level 2 (P): 맥락 추가 + 스킬화
자동 생성 보고서에 내러티브(왜, 어떤 의미, 어떤 결정 필요)를 추가하세요. 반복 사용을 위해 SKILL.md로 저장하세요.

### Level 3 (L): 멀티 프로젝트 보고서
2개 이상의 프로젝트 상태를 하나의 포트폴리오 보고서로 통합하세요. 프로젝트 간 리소스 충돌, 의존성, 전체 포트폴리오 건강도를 포함하세요.
