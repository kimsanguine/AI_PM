# 슬래시 커맨드: 반복 워크플로우를 원클릭으로

> **핵심 개념**: PM의 반복되는 업무를 `.claude/commands/` 폴더의 마크다운 파일로 정의하고, Claude Code에서 `/명령어`로 실행하는 가장 강력한 생산성 도구

---

## 슬래시 커맨드란?

### 정의
```
.claude/commands/ 폴더 내 마크다운 파일 → /파일명 으로 원클릭 실행
```

슬래시 커맨드는 Claude Code의 가장 강력한 워크플로우 자동화 기능입니다. 매번 같은 프롬프트를 반복 입력하는 대신, 표준화된 명령어로 즉시 실행할 수 있습니다.

### PM이 얻는 것
- **시간 절감**: 15분 프롬프트 작성 → 10초 커맨드 실행
- **일관성**: 같은 절차로 동일한 품질의 결과 생성
- **추적성**: 어떤 커맨드로 생성된 결과인지 명확
- **팀 공유**: 팀 전체가 같은 워크플로우 사용 가능

---

## PM을 위한 6가지 핵심 슬래시 커맨드

### 1. `/today` — 일일 브리핑

**목적**: 아침 출근 후 5분 안에 오늘의 우선순위 파악

```markdown
# .claude/commands/today.md

오늘의 브리핑을 생성해줘.

절차:
1. tasks/ 폴더에서 "오늘 마감" 또는 "과거 미완료" 태스크 목록화
2. status/ 에서 최근 주간 보고서 읽고 이번 주 맥락 파악
3. research/digest/ 에서 어제 뉴스 다이제스트 요약
4. content/ 에서 작성 중인 콘텐츠 상태 확인

출력 형식:
## 🎯 오늘의 우선순위 (최대 3개)
- [우선순위 1]: 마감일시 및 기대효과
- [우선순위 2]: 마감일시 및 기대효과
- [우선순위 3]: 마감일시 및 기대효과

## 📋 마감 임박 태스크 (2일 이내)
- [태스크 1]: D-1 | 진행률 | 블로커 있는가?

## 📊 이번 주 진행 현황
- 계획 대비 진행률
- 이번 주 주요 변수

## 📰 어제 핵심 뉴스 (Top 3)
- 뉴스 제목 | 왜 PM으로서 중요한가?
```

**사용 예시**:
```bash
$ /today
```

**PM 판단 포인트**:
- 우선순위 3개가 현실적인가? (5개 이상 = 방향 재정렬 신호)
- 블로커가 있는 태스크는 즉시 처리해야 하는가?
- 어제 뉴스 중 이번 주 전략에 영향을 주는 것이 있는가?

---

### 2. `/prd` — PRD 자동 생성

**목적**: 새 기능 또는 프로젝트의 Lean PRD 1시간 안에 초안 작성

```markdown
# .claude/commands/prd.md

기능명: $ARGUMENTS

Lean PRD를 생성해줘. 다음 구조로 작성:

## 1️⃣ 왜? (Why) - 1줄 정의
[시장 상황] 때문에 [대상 사용자]는 [문제]를 겪는다.
→ 해결책: [솔루션]

## 2️⃣ 성공 기준 (Metrics - OMTM)
One Metric That Matters: [지표명] → [현황] → [목표]
예: 월간 활성사용자 | 현황: 5,000명 | 목표: 10,000명 (8주)

## 3️⃣ 핵심 시나리오 (3개만)
### 시나리오 A: [사용자 직무]
- 입력: [사용자가 하는 일]
- 기대 결과: [우리 솔루션 이후]

## 4️⃣ 제약 (Constraints)
- 기술 제약:
- 비즈니스 제약:
- 시장 제약:

## 5️⃣ POC (Proof of Concept)
최소 검증 범위: [가장 핵심적인 1개 시나리오만]

---

이 PRD를 Notion에 게시할까요?
```

**사용 예시**:
```bash
$ /prd "AI 기반 자동 보고서 생성"
```

**PM 판단 포인트**:
- OMTM이 명확한가? (여러 지표 = 전략 흔들림)
- 3개 시나리오가 사용자의 진짜 문제를 해결하는가?
- 이 PRD를 엔지니어에게 보낼 수 있을 정도인가?

---

### 3. `/status` — 주간 상태 보고서

**목적**: 금요일 오후, 이번 주 성과를 정량적으로 정리

```markdown
# .claude/commands/status.md

이번 주 상태 보고서를 생성해줘.

데이터 수집:
1. tasks/ 에서 완료된 태스크 + 진행 중 태스크 + 미완료 태스크 분류
2. prd/ 에서 이번 주 생성된 PRD 및 업데이트된 PRD 목록
3. content/ 에서 작성/게시된 콘텐츠 목록
4. metrics.md 에서 주간 KPI 변화

출력 형식:

## 📈 주간 성과 (Week of 2026-02-XX)

### ✅ 완료 (주간)
- [완료된 태스크들: 총 X개]

### 🔄 진행 중 (이월 예정)
- [진행 중인 태스크들: 총 X개]

### 🚨 블로커
- [블로커 1]: 원인 | 영향 범위 | 해결 예정일
- [블로커 2]: 원인 | 영향 범위 | 해결 예정일

### 📊 핵심 지표
| 지표 | 지난주 | 이번주 | 변화 |
|------|--------|--------|------|
| MAU | 50K | 52K | +4% |
| 콘텐츠 발행수 | 8 | 12 | +50% |

### 🎯 다음 주 계획
1. [우선순위 1]
2. [우선순위 2]
3. [우선순위 3]

---

이 보고서를 Slack #status-report 채널에 공유할까요?
```

**사용 예시**:
```bash
$ /status
```

**PM 판단 포인트**:
- 완료 대비 진행 중 비율이 건강한가? (완료 < 진행 중 = 일정 위험)
- 블로커가 2개 이상이면 다음 주 계획 재조정 필요한가?
- 핵심 지표가 목표 방향과 일치하는가?

---

### 4. `/competitor` — 경쟁사 5축 분석

**목적**: 경쟁 전략 회의 전, 90분 안에 경쟁사 분석 완료

```markdown
# .claude/commands/competitor.md

경쟁사명: $ARGUMENTS

웹 검색을 통해 경쟁사를 5축으로 분석해줘.

## 🔍 5축 분석

### 1️⃣ Product (제품)
- 핵심 기능 Top 3
- 최근 업데이트 (지난 3개월)
- 차별화 포인트

### 2️⃣ Pricing (가격)
- 가격 정책 (Free/Pro/Enterprise)
- 타겟 고객층별 CAC
- 예상 ARR (Available data)

### 3️⃣ Positioning (포지셔닝)
- 마케팅 메시지
- 타겟 고객
- Brand Tone

### 4️⃣ Performance (성과)
- 사용자 규모 (추정)
- 성장률 추세
- 시장 점유율 (가능한 범위)

### 5️⃣ People (사람)
- CEO/Founder 배경
- 주요 투자자
- 팀 규모

## 💡 전략적 인사이트

### 가설 3개
1. [가설]: 근거
2. [가설]: 근거
3. [가설]: 근거

### Moat 분석 (경쟁 해자)
- 이 경쟁사가 지킬 수 있는 것: [기술/네트워크/브랜드/규제]
- 우리가 공략할 수 있는 것: [시장 틈새/새로운 기술/다른 세그먼트]
```

**사용 예시**:
```bash
$ /competitor "OpenAI"
$ /competitor "Anthropic"
```

**PM 판단 포인트**:
- 이 경쟁사가 우리의 진정한 경쟁자인가? (시장/고객/가격이 겹치는가?)
- 가설 3개 중 우리가 검증해야 할 것은? (리서치 계획 수립)
- 해자 분석이 우리의 전략에 영향을 주는가?

### 5. `/linkedin` — LinkedIn 포스트 초안

**목적**: 주간 1-2회 콘텐츠 마케팅 포스트, 30분 안에 3가지 옵션 생성

```markdown
# .claude/commands/linkedin.md

주제: $ARGUMENTS

LinkedI 포스트 초안 3가지를 생성해줘. 각 포스트는:
1. 톤앤매너는 llm-context/brand/voice.md 를 참고
2. 구조: 질문/상황 → 경험 기반 관찰 → 인사이트 → 소프트 CTA
3. 길이: 150-300자 (LinkedIn 알고리즘 최적화)

---

## 옵션 1: [타입] 포스트

[개인 경험/관찰을 시작으로 하는 질문]

"..." [짧은 스토리라인]

→ [우리의 인사이트]

→ [호출 액션] (댓글/공유 유도)

---

## 옵션 2: [타입] 포스트

[다른 앵글의 콘텐츠]

---

## 옵션 3: [타입] 포스트

[또 다른 앵글]

---

이 중 어느 포스트를 게시할까요?
```

**사용 예시**:
```bash
$ /linkedin "AI 시대의 PM 역할 변화"
$ /linkedin "데이터 기반 의사결정의 함정"
```

**PM 판단 포인트**:
- 3개 옵션 중 우리 브랜드 톤과 가장 맞는 것은?
- 이 포스트가 타겟 고객(CEO/PM)의 공감을 얻을까?
- CTA가 자연스럽고 강압적이지 않은가?

---

### 6. `/agent` — Vibe Agenting 프로토타입

**목적**: 새로운 AI Agent 아이디어를 TDD로 빠르게 검증

```markdown
# .claude/commands/agent.md

Agent 아이디어: $ARGUMENTS

다음 순서로 Agent 프로토타입을 구축해줘.

## Step 1: 문제 정의
- 이 Agent가 해결해야 할 문제는?
- 성공의 정의는? (1-2개의 구체적인 테스트 케이스)

## Step 2: 테스트 케이스 (Eval First)
```
테스트 케이스 1:
- 입력: [Agent에 주는 명령]
- 기대 결과: [이상적인 아웃풋]
- 성공 기준: [어떻게 성공을 판단할 것인가]
```
## Step 3: Agent Instructions (마크다운)
Agent의 역할, 목표, 사용할 도구, 제약을 명확히 정의

## Step 4: 프로토타입 코드
가장 간단한 형태의 작동하는 코드 (최소 50줄, 최대 100줄)

## Step 5: 테스트 & 결과
Step 2의 테스트 케이스들을 프로토타입에 실행 → 통과/실패 기록

## 다음 스텝
- 통과: 실제 구현 로드맵 수립
- 실패: 어느 부분이 막혔는가? → Agent Instructions 수정 → 재테스트
```

**사용 예시**:
```bash
$ /agent "경쟁 분석을 자동으로 수행하는 에이전트"
```

**PM 판단 포인트**:
- 이 Agent가 정말 필요한가? (수동으로는 불가능한 규모인가?)
- 테스트 케이스를 통과하는 프로토타입이 구축되었는가?
- 이것을 실제 제품으로 확대하려면 어떤 투자가 필요한가?

## 슬래시 커맨드 작성 팁

### 1. 구체적인 출력 형식 지정
```markdown
❌ 나쁜 예: "PRD를 작성해줘"

✅ 좋은 예: "다음 마크다운 구조로 PRD를 작성해줘:
## 왜? (Why)
## 성공 기준
## 핵심 시나리오 (3개)
..."
```

### 2. 파일 경로 명시
```markdown
✅ "tasks/ 폴더의 모든 .md 파일을 읽고..."
✅ "status/ 의 최근 주간 보고서를 찾아서..."
```

### 3. $ARGUMENTS 활용 (동적 입력)
```bash
# $ARGUMENTS 없음: /today 는 항상 고정된 결과
# $ARGUMENTS 있음: /prd "기능명" 으로 유연하게 대응
```

### 4. 후속 액션 질문 포함
```markdown
✅ "이 보고서를 Slack에 공유할까요?"
✅ "이 PRD를 팀에 공개할까요?"

→ PM이 최종 판단하는 구조 (자동화 ≠ 무조건 실행)
```

## Before & After: 주간 워크플로우 비교

### 🔴 Before: 수동 워크플로우 (매주 3-4시간 소요)

```
Monday 09:00
- [수동 입력] 오늘 할 일 목록 작성 (15분)
- 슬랙, 노션, 태스크 앱을 번갈아가며 확인

Tuesday 14:00
- [수동 입력] "좋은 PRD를 작성하는 프롬프트"를 검색 (5분)
- 프롬프트를 커스터마이징 (10분)
- Claude에 복사-붙여넣기 (5분)
- 출력 결과를 노션에 수동 정리 (15분)

Wednesday 11:00
- [수동 입력] 경쟁사 분석 프롬프트 작성 (15분)
- 웹 검색 링크들을 수동으로 정리 (20분)

Friday 17:00
- [수동 입력] 주간 보고서를 엑셀에 손으로 작성 (45분)
```

### 🟢 After: 슬래시 커맨드 워크플로우 (매주 30분 소요)

```
Monday 09:00
$ /today
→ 2분 안에 오늘의 우선순위 파악

Tuesday 14:00
$ /prd "새 기능명"
→ 5분 안에 PRD 초안 + 게시 확인

Wednesday 11:00
$ /competitor "경쟁사명"
→ 10분 안에 5축 분석 + 전략 인사이트

Friday 17:00
$ /status
→ 5분 안에 주간 보고서 자동 생성 + Slack 공유 확인
```

**시간 절감**: 3.5시간 → 0.5시간 (86% 단축)
**품질**: 표준화된 형식으로 일관성 상승
**재사용성**: 같은 커맨드를 팀과 공유 가능

## PM 역할 확장 포인트

### 슬래시 커맨드는 "자동화"가 아니라 "확장"

| 구분 | Before (전통 PM) | After (슬래시 커맨드) |
| --- | --- | --- |
| 시간 투입 | 매주 3-4시간 | 매주 30분 |
| 판단 횟수 | 적음 (바쁘면 스킵) | 많음 (빨라서 자주) |
| 데이터 기반 | 약함 (메모리 의존) | 강함 (모든 결과 기록) |
| 팀 정렬 | 약함 (일관성 없음) | 강함 (표준화) |

### PM이 판단해야 할 것

```
❌ 무조건 자동화할 것
- 민감한 의사결정 (전략, 인원 조정, 배포 시점 등)
- 정성적 판단이 필요한 것 (고객 피드백 해석 등)

✅ 자동화해야 할 것
- 반복적인 정보 수집 (/today, /status)
- 표준화된 산출물 (/prd, /competitor, /linkedin)
- 빠른 프로토타이핑 (/agent)

🔀 슬래시 커맨드가 돕는 것
- "판단을 위한 정보 수집"을 빠르게
- PM이 "판단 자체"에 더 많은 시간을 쓸 수 있게
```

### 성숙도 단계

```
Level 1 (이번 주): /today, /prd 를 사용해보기
Level 2 (다음 주): 팀의 피드백을 받아 커맨드 개선
Level 3 (한 달): 팀 전체가 같은 커맨드 사용 (일관성 확보)
Level 4 (분기): 새로운 워크플로우 발견 → 새 커맨드 추가
```

## 실습 과제

### Level 2: 나만의 커맨드 만들기

```
1단계: /today 와 /prd 를 본인 프로젝트에 맞게 커스터마이징
  - task 폴더의 실제 경로명으로 수정
  - 본인이 실제로 사용할 output 형식으로 수정

2단계: 3일 연속 /today 실행
  - Monday: 주간 우선순위 파악
  - Tuesday: 어제 결정사항이 오늘 일정에 반영되는가?
  - Wednesday: 주중 변화가 금요일 보고서에 영향을 주는가?

3단계: CLAUDE.md 에 피드백 루프 기록
  - 각 /today 실행 후, "PM 판단 결과" 기록
  - 예: "블로커 1개 발견 → 즉시 처리 → 화요일 완료"
  - 예: "우선순위 변경 → /prd 추가 실행 → 새 기능 검토"

4단계 (선택): 팀과 공유
  - 본인의 커맨드를 README로 정리
  - 팀원 1명에게 /prd 시연해보기
```

### 성공 기준
- ✅ /today 를 월-수 3회 실행하고 각각 "PM 판단"을 기록
- ✅ /prd 로 1개 이상의 실제 기능 PRD 작성 완료
- ✅ CLAUDE.md 에 "이 커맨드로 절감된 시간" 정량화

## 자주 묻는 질문 (FAQ)

### Q: 슬래시 커맨드가 항상 좋은 결과를 주나요?
**A**: 아니요. 입력이 정확할수록 좋은 결과가 나옵니다.
- `/prd "새 기능"` ← 불명확 (실패율 높음)
- `/prd "AI 기반 자동 보고서 생성 기능 (타겟: 마케팅팀, 8주 POC)"` ← 명확 (성공율 높음)

### Q: 팀원들과 다른 커맨드를 쓰면?
**A**: 산출물의 형식과 품질이 달라집니다. 팀 내 표준화가 중요합니다.
- 권장: 팀 리드가 1-2개 커맨드를 정의 → 팀 전체 사용
- 금지: 각자 다른 프롬프트 사용 (일관성 깨짐)

### Q: 커맨드를 수정해야 할 때는?
**A**: 매주 피드백을 받아 개선하세요.
- 월요일: 지난주 커맨드 결과 리뷰
- 화요일: 개선사항 적용 (마크다운 수정)
- 수요일부터: 새로운 커맨드로 실행

## 마치며

슬래시 커맨드의 진정한 가치는 **"시간 절감"이 아니라 "의사결정 속도"입니다.**

- Before: 정보 수집 (3시간) → 의사결정 (30분)
- After: 정보 수집 (10분) → 의사결정 (1시간)

PM은 더 많은 정보로 더 빠르게, 더 좋은 판단을 할 수 있습니다.

다음 파트 "데이터 기반 의사결정"에서는, 이렇게 빠르게 수집한 데이터를 어떻게 의사결정으로 연결할 것인지 다룹니다.
