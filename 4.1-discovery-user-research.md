# 유저 리서치 데이터 분석

## 개요
Discovery 단계에서 PM의 역할은 문제를 올바르게 정의하는 것입니다. Claude Code는 대량의 정성/정량 데이터를 빠르게 구조화하여 PM이 **패턴을 발견하고 의사결정하는 데 집중**하도록 지원합니다.

이 모듈에서는:
- CSV/설문 데이터의 빠른 세그멘테이션
- 인터뷰 노트의 대규모 텍스트 분석
- 리서치 결과의 자동 종합

을 통해 **PM이 직접 1차 분석을 수행**할 수 있는 방법을 다룹니다.

## 섹션 1: 설문 데이터 분석 - 세그멘트 발견

### 상황
마케팅 팀이 847개 사용자 설문 응답을 받았지만, 정리되지 않은 상태입니다.
PM은 이 데이터에서 사용자 그룹을 파악하고, 각 그룹의 특성을 이해해야 합니다.

### Claude Code 워크플로우

```bash
cc claude code

> @user-survey-results.csv 이 설문 데이터를 분석해줘.
  1. 응답자를 행동 패턴 기반으로 3~5개 세그먼트로 분류
  2. 세그먼트별 만족도, 주요 불만, 사용 빈도 요약
  3. 가장 큰 기회 영역을 근거와 함께 제시
```


## 섹션 2: 인터뷰 노트 통합 분석

### 상황
사용자 리서처가 12건의 심층 인터뷰를 진행했습니다.
각각 30분~1시간짜리 노트 파일로 저장되어 있지만, 아직 정리되지 않았습니다.
PM은 공통 패턴과 Jobs-to-be-done을 파악해야 합니다.

### Claude Code 워크플로우

```bash
> @user-interviews/ 이 폴더의 인터뷰 12건을 분석해서:
  1. 공통 Pain Point 상위 5개 (빈도순)
  2. 각 Pain Point별 대표 인용문 3개
  3. Jobs-to-be-done 프레임워크로 정리
  4. 각 Job 별 현재 솔루션과의 갭 분석
```

### Claude의 응답 (실제 터미널 아웃풋)

```
Claude: 12건 인터뷰(총 약 18,000단어)를 분석했습니다.

## Pain Point 분석 (빈도순)

### 1. "할 일이 여러 도구에 흩어져 있다" (12명 중 11명, 92%)

  매일 아침 체크리스트 만들어야 해"

### 2. "우선순위를 자주 바꿔서 집중하기 힘들다" (11명, 92%)

- 인터뷰이 E: "매니저가 항상 '이게 더 급해'라고 해서...

### 3. "개인 일과 팀 일의 경계가 불명확하다" (9명, 75%)
- 인터뷰이 D: "내 투두와 팀 목표가 어떻게 연결되는지 몰라"

### 4. "완료했는지 못했는지 추적이 어렵다" (8명, 67%)
- 인터뷰이 F: "할 일 목록에 있는 것 중 뭐가 실제로 완료된 건지...
  수동으로 업데이트하려니 자꾸 놓쳐"
- 인터뷰이 L: "상태 업데이트가 너무 많아. 대신 해주는 게 있으면 좋겠어"

- 인터뷰이 C: "다른 팀과 일할 때 권한이 없어서 항상 관리자 거쳐야 해"
- 인터뷰이 I: "누가 뭘 승인하는지, 다음 단계가 뭔지 명확하지 않아"

## Jobs-to-be-Done 분석

### Job #1: "나의 일일 포커스를 유지하고 싶다"
- Functional Job: 하루 중 해야 할 일을 한곳에서 관리
- Emotional Job: "내가 의존한다"는 신뢰감
- Social Job: 팀에 진행 상황을 보이고 신뢰받기
- 현재 솔루션의 갭: 여러 도구 → PM이 매일 취합 필요

### Job #2: "팀 목표와 나의 일을 연결하고 싶다"
- Functional Job: OKR/목표를 분해한 일일 태스크 시스템
- Emotional Job: "나의 일이 의미 있다"는 느낌
- Social Job: 팀 진행 상황을 함께 본다
- 현재 솔루션의 갭: 목표는 따로, 실행은 따로

### Job #3: "상태 변화를 자동으로 추적하고 싶다"
- Functional Job: 완료한 일을 체크만 하면 상태 자동 반영
- Emotional Job: "내가 진행 중이라는 걸 증명하고 싶다"
- Social Job: 실시간 팀 진행도 가시화
- 현재 솔루션의 갭: 수동 업데이트 → 자주 최신이 아님
```

## PM 역할 확장 포인트

### Before: 기존 워크플로우
```
Day 1-2: 리서처에게 "설문 데이터와 인터뷰 노트 분석해줄 수 있어?" 요청
Day 8-9: 보고서 수령
Day 10-12: PM이 보고서 읽고 해석
Day 13-14: 리서처와 함께 리뷰 미팅

총 소요 시간: 약 2주
PM의 역할: 결과 수신자 → 의사결정자
```

### After: Claude Code 활용 워크플로우
```
시간 0: PM이 수집된 데이터(CSV, 인터뷰 노트) 준비
시간 1: Claude Code로 1차 분석 수행 → 세그먼트, Pain Point 추출
시간 2: PM이 결과 검토하고 추가 질문 2-3개 던지기
  예) "Churning User 중에서 우리가 재활성화할 수 있는 기준은?"
시간 3-4: 리서처와 심화 논의 (검증, 액션 아이템 정의)

총 소요 시간: 약 4-6시간 (집중 1일)
PM의 역할: 1차 분석 주인 → 가설 검증자 → 의사결정자
```

### 핵심 변화
- **속도**: 2주 → 1일
- **PM의 주인의식**: "리서처의 보고서" → "내 분석"
- **품질**: 표면적 요약 → 깊이 있는 대화로 발전

### 필요한 스킬 전환
- **이전**: "분석 보고서를 어떻게 읽을 것인가"
- **이후**: "Claude와의 대화를 통해 어떤 추가 가설을 만들 것인가"

### 🤔 PM 딜레마

**상황**: Claude 분석 결과, Casual User(41%) 활성화가 ROI 1순위라고 나왔다. 그런데 CEO는 "Power User 프리미엄화를 이번 분기에 반드시 해야 한다"고 강하게 주장한다. 이사회에서 매출 성장 목표를 높게 잡았기 때문이다.

데이터는 Casual User 세그먼트의 활성화를 최우선 추천했지만, 경영진은 높은 수익화 가능성을 가진 Power User 전략을 고집한다. 두 접근법 모두 근거 있지만, 제한된 자원으로는 둘 다 할 수 없다. 데이터 기반 제안과 경영진의 전략적 판단이 정면 충돌한 것이다.

**생각해볼 질문:**
- 데이터 기반 제안과 경영진의 전략적 판단이 충돌할 때, PM은 어떤 근거로 자신의 입장을 정해야 하는가? 데이터만 신뢰할 수 있는가?
- "두 마리 토끼를 모두 잡겠다"는 타협이 오히려 더 위험한 경우는 언제인가?
- 만약 CEO의 판단이 맞을 확률이 40%라면, 그래도 따라야 하는가?

## ⚠️ 이럴 때는 이렇게

실전에서 Claude와 리서치 데이터를 분석하다 보면 예상치 못한 문제들이 발생합니다. 이런 상황들을 어떻게 대처하는지 알아봅시다.

### **상황 1: CSV 파일의 한글이 깨져서 Claude가 데이터를 잘못 분석하는 경우**

**증상:**
- Claude의 분석 결과에 세그먼트 이름이 "???" 또는 이상한 문자로 표시됨
- "주요 불만" 섹션이 비어있거나 엉뚱한 내용이 추출됨
- "이 파일을 읽을 수 없습니다"라는 에러 메시지 출력

**원인:**
- CSV 파일이 EUC-KR (Windows 기본 인코딩)로 저장되어 있는데, Claude가 UTF-8로 해석하려고 시도
- 한글 문자가 바이트 레벨에서 손상되어 Claude가 데이터를 올바르게 파싱하지 못함
- 설문 도구에서 한글을 포함한 CSV를 다운로드할 때 발생하는 흔한 문제

**해결 방법:**

먼저 Claude에게 직접 문제를 알립니다:
```bash
> 이 파일 인코딩이 EUC-KR인 것 같아. UTF-8로 변환하고 다시 분석해줘.
```

또는 터미널에서 직접 변환한 후 업로드:
```bash
$ iconv -f EUC-KR -t UTF-8 user-survey.csv > user-survey-utf8.csv

$ cc claude code

> @user-survey-utf8.csv 이제 올바르게 분석해줄 수 있어?
```

**예방 팁:**
- 설문 도구에서 CSV 다운로드할 때 "UTF-8 인코딩으로 저장" 옵션이 있는지 확인
- Google Sheets나 Airtable 같은 클라우드 도구를 사용하면 인코딩 문제 없음
- 처음 몇 줄만 Claude에 보여주고 "인코딩 확인"을 요청해볼 수 있음

### **상황 2: Claude가 세그먼트를 너무 많이(또는 너무 적게) 만들어서 의미 없는 경우**

**증상:**
- Claude가 10개의 세그먼트를 만들었는데, 일부는 응답자 수가 1~2명 수준
- 또는 반대로 2개의 매우 큰 세그먼트만 있어서 차이점을 찾기 어려움
- "이 세그먼트들이 정말 의미 있는 그룹인가?"라는 의문이 들 때

**원인:**
- Claude가 주어진 요청("3~5개 세그먼트")을 무시하고 데이터가 자연스럽게 나뉘는 대로 분류
- PM이 "세그먼트"를 정의할 때 구체적인 기준(예: 행동 패턴, 이탈 신호)을 제시하지 않음
- 데이터에 이상치가 많거나, 분포가 치우쳐 있는 경우

**해결 방법:**

Claude에게 구체적인 기준과 최소 표본을 재요청합니다:
```bash
> 세그먼트가 너무 많아. 행동 패턴 기준으로 3~4개로 다시 묶어줄 수 있어?
  조건:
  1. 각 세그먼트는 최소 5% 이상이어야 함
  2. 행동 차이가 명확해야 함 (예: 사용 빈도, 이탈 신호)
  3. 세그먼트 이름은 PM이 리더십에 설명하기 쉬운 이름으로
```

또 다른 방법으로는 사전에 기준을 명시합니다:
```bash
> @survey-data.csv 이 데이터를 다음 기준으로 3개 세그먼트로 분류해줘:
  - 활성 사용자: 주 5회 이상 사용
  - 일반 사용자: 주 1~4회 사용
  - 유휴 사용자: 월 1회 이하
```

**PM의 판단:**
- 실제로 세그먼트가 너무 많으면 "각 세그먼트가 정말 다른가?"를 물어봅니다
- 세그먼트 간 특성 차이(만족도, 이탈율 등)를 확인하고, 중복되는 그룹을 합쳐도 되는지 판단

### **상황 3: Claude가 상관관계를 인과관계로 해석하는 경우**

**증상:**
- Claude: "모바일 사용자의 이탈률이 높으므로, 모바일 환경이 이탈의 주요 원인입니다"
- 결론: "모바일 UI를 개선해야 함"
- 하지만 실제로는 모바일 사용자가 대부분 "신규 사용자"일 뿐, 모바일 자체가 문제가 아닐 수도 있음

**원인:**
- Claude는 데이터 기반으로만 분석하므로, 상관관계만 볼 수 있음
- "모바일 사용자 = 높은 이탈률"이라는 패턴은 보지만, 그 뒤의 컨텍스트(신규 vs 기존 사용자 구성 차이)는 모름
- Simpson's Paradox: 전체 데이터에서는 한 패턴이 보이지만, 세부 그룹으로 나누면 반대 결과가 나올 수 있음

**해결 방법:**

PM이 다음과 같이 질문을 던져야 합니다:
```bash
> 모바일 사용자의 이탈률이 높다고 했는데,
  모바일 사용자 중에서도 "신규 vs 기존 사용자"로 다시 분석해줄 수 있어?
  혹시 모바일 사용자가 대부분 신규라서 그런 건 아닐까?
```

또는 구체적인 검증을 요청:
```bash
> "모바일이 이탈 원인"이라는 주장을 검증하기 위해
  다음을 확인해줘:
  1. 신규 모바일 사용자 vs 신규 웹 사용자의 이탈률
  2. 6개월 이상 활성 사용자 중 모바일 vs 웹 비교
  3. 모바일에서 사용 시간이 짧으면 정말 이탈하는가?
```

**PM의 역할:**
- Claude의 분석 결과를 보고 "이게 정말 원인일까, 아니면 다른 변수 때문일까?"를 항상 의심
- 인과관계를 주장하기 전에 "세그먼트별 비교"를 추가로 요청
- 실제 인과관계를 확인하려면 "사용 이유"를 직접 묻는 정성 인터뷰가 필요함을 알기
