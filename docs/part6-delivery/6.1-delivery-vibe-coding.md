# 4.1 바이브 코딩: PM이 직접 프로토타입 만들기

## 바이브 코딩이란?

**바이브 코딩(Vibe Coding)**은 PM이 코드를 직접 작성하지 않고, 자신의 의도와 요구사항을 자연스러운 언어로 설명하면 Claude가 전체 구현을 담당하는 방식입니다.

핵심은 **"무엇을 만들 것인가"에 집중**하고, **"어떻게 구현할 것인가"는 Claude에 맡기는 것**입니다.

이를 통해 PM은:

- 프로토타입 제작의 신속성 확보 (당일 완성 가능)
- 개발자의 시간 확보 (반복 수정 없이 최종 결과물 전달)
- 아이디어 검증의 속도 향상

## 실전: 성격 유형 테스트 웹앱 프로토타입

### 1단계: 프로젝트 초기화

```bash
$ cd ~/my-prototype
$ mkdir personality-quiz
$ cd personality-quiz
$ claude
```

### 2단계: 요구사항 설명

Claude 대화창에서:

```
> 성격 유형 테스트 웹앱을 만들려고 해. Next.js로.
  - 5개의 질문에 답하면 4가지 유형 중 하나를 알려주는 앱이야
  - 각 질문은 라디오 버튼으로 선택 가능
  - 제출 후 결과 페이지에서 유형 설명과 특징을 보여줌
  - 결과를 공유할 수 있는 고유 URL을 생성
  - 디자인은 최신 Tailwind CSS 기반으로 깔끔하게
```

Claude의 응답:

```
성격 유형 테스트 웹앱을 설계하겠습니다.

프로젝트 구조:
- pages/index.tsx: 테스트 시작 페이지
- pages/quiz.tsx: 질문 페이지 (상태 관리)
- pages/result.tsx: 결과 페이지
- components/Question.tsx: 개별 질문 컴포넌트
- lib/personalities.ts: 성격 유형 데이터
- lib/scoring.ts: 점수 계산 로직

먼저 REQUIREMENTS.md를 생성하고, 프로젝트 초기화를 진행하겠습니다.
```

### 3단계: 요구사항 문서 생성 및 검증

Claude가 생성한 REQUIREMENTS.md 내용:

```markdown
# 성격 유형 테스트 - 요구사항 명세

## 기능 요구사항
1. 랜딩 페이지: 테스트 소개 및 시작 버튼
2. 퀴즈 페이지: 5개 질문, 각각 4가지 선택지
3. 결과 페이지: 유형명, 설명, 특징
4. 공유 기능: 결과 URL 생성 및 복사

## 성격 유형 정의
- Type A: 주도형 (Leadership)
- Type B: 공감형 (Empathy)
- Type C: 분석형 (Analysis)
- Type D: 실행형 (Action)

## 기술 스택
- Next.js 14
- React 18
- Tailwind CSS
- TypeScript
```

PM 검증:

```
> 좋아. 이 구조로 진행해줘. 초기 프로젝트 구성도 자동으로 해줄 수 있어?

Claude: 네, package.json, tsconfig.json, next.config.js를 설정하고
필요한 파일 구조를 모두 생성하겠습니다.
```

## 실전 심화: Pencil 연동으로 실제 디자인 만들기

### Pencil.dev란?

**Pencil.dev**는 AI 네이티브 프론트엔드 디자인 도구로, IDE(VS Code, Cursor) 내부에서 직접 디자인 작업을 할 수 있다. 핵심은 **MCP(Model Context Protocol)를 통해 Claude Code와 직접 연결**된다는 점이다.

**기존 디자인 도구와의 차이:**

|  | Figma | Canva | **Pencil.dev** |
| --- | --- | --- | --- |
| 인터페이스 | 웹 브라우저 | 웹 브라우저 | **IDE 내장 (VS Code/Cursor)** |
| 결과물 | 디자인 파일 (수동 핸드오프) | 이미지/PDF | **`.pen`****파일 → 코드 자동 생성** |
| 개발 연결 | Inspect/Dev Mode (별도) | 없음 | **Git 리포에 바로 커밋** |
| AI 연동 | 제한적 | 템플릿 기반 | **Claude Code MCP 직접 연동** |
| PM 적합도 | 디자이너 중심 | 마케팅 중심 | **PM의 바이브 코딩에 최적화** |

**PM에게 Pencil이 중요한 이유:**

1. 디자인 → 코드 전환의 "로스트 인 트랜슬레이션" 문제 해결
2. PM이 의도한 디자인이 **그대로 코드로 변환** (Figma ↔ 개발 핸드오프 불필요)
3. `.pen` 파일이 Git에 저장 → 디자인 변경도 버전 관리 가능

### 설정: Claude Code + Pencil MCP 연결

**Step 1: Pencil 설치**

VS Code 또는 Cursor에서 Pencil 확장 프로그램을 설치한다.

```bash
# VS Code에서
# Extensions (Ctrl+Shift+X) → "Pencil" 검색 → Install

# 또는 CLI로
code --install-extension pencil.pencil-dev
```

**Step 2: MCP 설정**

Claude Code의 MCP 설정에 Pencil 서버를 추가한다.

```json
// .claude/settings.json
{
  "mcpServers": {
    "pencil": {
      "command": "npx",
      "args": ["-y", "@anthropic-ai/pencil-mcp-server"]
    }
  }
}
```

**Step 3: 연결 확인**

```bash
$ claude
> /mcp

# Pencil MCP 서버가 연결되었는지 확인
# 사용 가능한 도구: read_design, update_design, create_component 등
```

### 실전 워크스루: 대시보드 UI 디자인 + 코드 생성

PM이 KPI 대시보드의 UI를 직접 설계하고, 바로 작동하는 코드로 변환하는 시나리오다.

**Step 1: 디자인 의도 설명**

```bash
> 우리 제품의 KPI 대시보드를 Pencil로 디자인해줘.

  포함할 요소:
  - 상단: 핵심 지표 카드 4개 (MAU, MRR, Churn Rate, NPS)
  - 중간: 주간 트렌드 차트 (라인 그래프)
  - 하단: 최근 알림 테이블

  스타일:
  - 다크 모드 테마
  - 카드에 전주 대비 변화율 표시 (↑ 녹색, ↓ 빨간색)
  - 모바일 반응형
```

**Claude의 응답:**

```
Claude: Pencil 캔버스에 대시보드를 설계하겠습니다.

✓ dashboard.pen 파일 생성
✓ KPI 카드 4개 배치 (MAU: 12.5K ↑3.2%, MRR: $250K ↑1.8%,
  Churn: 4.2% ↓0.3%, NPS: 72 ↑5)
✓ 주간 트렌드 차트 영역 설정
✓ 알림 테이블 레이아웃 완성
✓ 다크 모드 컬러 팔레트 적용

VS Code에서 dashboard.pen 파일을 열면 디자인을 확인할 수 있습니다.
```

**Step 2: 디자인 리뷰 & 수정**

PM이 VS Code에서 `.pen` 파일을 열어 디자인을 확인한 후:

```bash
> dashboard.pen을 확인했어. 수정사항:
  1. KPI 카드 순서를 MRR → MAU → NPS → Churn으로 변경
     (비즈니스 중요도 순)
  2. 차트에 "목표선"을 점선으로 추가
  3. 알림 테이블에 "심각도" 컬럼 추가 (🔴🟡🟢)
  4. 카드 클릭 시 해당 지표의 상세 뷰로 이동하는 인터랙션

Claude: dashboard.pen을 업데이트했습니다.

✓ KPI 카드 순서 변경 (MRR first)
✓ 목표선 추가 (점선, #FFD700)
✓ 심각도 컬럼 추가 with 색상 아이콘
✓ 카드 클릭 인터랙션 정의 (onClick → detail view)
```

**Step 3: 디자인 → 코드 자동 변환**

이것이 Pencil의 핵심 가치다. 완성된 디자인에서 바로 작동하는 React 코드를 생성한다.

```bash
> dashboard.pen 디자인을 React + Tailwind 코드로 변환해줘.
  Next.js 프로젝트 구조에 맞춰서.

Claude: Pencil 디자인을 기반으로 코드를 생성합니다.

✓ components/Dashboard.tsx 생성
✓ components/KPICard.tsx 생성
✓ components/TrendChart.tsx 생성
✓ components/AlertTable.tsx 생성
✓ pages/dashboard.tsx 라우트 설정
✓ 모든 스타일은 Tailwind 클래스로 적용

디자인과 코드가 pixel-perfect으로 일치합니다.
npm run dev로 확인해보세요.
```

**Step 4: 실시간 디자인-코드 동기화**

Pencil의 또 다른 강점: 디자인을 수정하면 코드가 자동으로 업데이트된다.

```bash
> dashboard.pen에서 KPI 카드의 모서리를 더 둥글게 (border-radius 16px),
  그림자를 좀 더 강하게 해줘. 코드에도 자동 반영돼?

Claude: Pencil 디자인을 수정하고 코드를 동기화합니다.

✓ dashboard.pen: KPI 카드 border-radius 16px 적용
✓ dashboard.pen: box-shadow 강도 증가
✓ components/KPICard.tsx: rounded-xl → rounded-2xl 자동 반영
✓ components/KPICard.tsx: shadow-md → shadow-xl 자동 반영

디자인 변경 → 코드 반영이 자동으로 완료되었습니다.
```

### 🎯 PM 판단 포인트: Pencil 활용 시

**언제 Pencil을 쓰고, 언제 안 쓰는가?**

| 상황 | Pencil 사용 | Figma/기존 도구 |
| --- | --- | --- |
| 프로토타입 빠르게 만들기 | ✅ 최적 | ❌ 핸드오프 시간 |
| 디자이너와 협업하는 최종 제품 | ❌ 디자이너 도구 존중 | ✅ 디자이너 중심 |
| 내부 대시보드/어드민 도구 | ✅ PM이 직접 가능 | ❌ 과도한 프로세스 |
| 마케팅 랜딩 페이지 | ✅ 빠른 A/B 테스트 | ⚠️ 브랜드 가이드 확인 |
| 디자인 시스템 구축 | ❌ 전문가 필요 | ✅ 디자이너 영역 |

**PM의 핵심 원칙**: Pencil은 **"디자이너를 대체하는 도구"가 아니라, "PM이 아이디어를 빠르게 시각화하고 검증하는 도구"**다. 최종 제품 디자인은 디자인팀과 협업해야 하지만, 방향 검증과 프로토타입은 PM이 직접 할 수 있다.

### Pencil 워크플로우 요약

```
PM의 의도 설명 (자연어)
    ↓
Claude가 .pen 파일 생성 (Pencil MCP)
    ↓
PM이 VS Code에서 디자인 리뷰
    ↓
수정사항 피드백 (자연어)
    ↓
Claude가 디자인 업데이트 + 코드 동기화
    ↓
npm run dev로 즉시 확인
    ↓
스크린샷 피드백 → 반복
```

기존 바이브 코딩이 "코드를 직접 작성하지 않는 것"이었다면, Pencil 연동은 여기에 "디자인도 직접 하지 않는 것"을 추가한다. PM은 **의도만 설명**하면, 디자인과 코드가 동시에 완성된다.

## PM 역할 확장 포인트

### BEFORE: 전통적 방식

```
PM이 요구사항 정의
  ↓
개발자에게 설계 문서 전달
  ↓
개발자 구현 (3~5일)
  ↓
첫 번째 리뷰 회의 (1~2일 대기)
  ↓
수정 요청 및 구현 (2~3일)
  ↓
최종 프로토타입 완성 (총 1~2주)
```

### AFTER: AI 네이티브 PM

```
PM이 의도 설명 및 스크린샷 피드백
  ↓
Claude가 즉시 구현
  ↓
PM이 실시간으로 반복 수정
  ↓
당일 완성된 프로토타입 획득
```

**시간 절감**: 1~2주 → 4시간
**PM의 역할 확장**: 수동 대기 → 능동적 프로토타입 검증 및 의사결정

### 🤔 PM 딜레마

**상황**: PM이 Claude와 바이브 코딩으로 프로토타입을 완성했다. 스테이크홀더들의 반응이 뜨겁다. CTO가 "이거 그냥 프로덕션으로 올리자"고 한다. 하지만 Claude가 만든 코드는 에러 처리, 보안, 성능 최적화가 빠져있다. 엔지니어링팀은 "처음부터 다시 만들어야 한다"고 한다.

PM이 직접 코드를 만들 수 있게 되면서 새로운 딜레마가 생겼다. 프로토타입의 진정한 역할이 무엇인지, 그리고 엔지니어링 팀과의 관계에 어떤 영향을 미칠지 고민해야 한다.

**생각해볼 질문:**

- 프로토타입의 역할은 "검증"인가, "출발점"인가? 프로토타입을 프로덕션 코드로 승격시키면 어떤 문제가 생기는가?
- "작동하는 프로토타입"이 오히려 올바른 엔지니어링을 방해하는 경우는 어떻게 대응하는가? 엔지니어링팀의 합리적 우려를 어떻게 설득할 것인가?
- PM이 "코드를 만들 수 있게 되었다"는 것이 엔지니어와의 관계에 어떤 영향을 미치는가? 긍정적 시나리오(빠른 반복, 신뢰)와 부정적 시나리오(영역 침범, 품질 저하) 모두 생각해보자.

## ⚠️ 이럴 때는 이렇게

바이브 코딩으로 프로토타입을 만들 때도 예상치 못한 문제들이 발생합니다. 실전 상황과 해결책을 알아봅시다.

### **상황 1: Claude가 만든 코드가 동작하지 않거나 에러가 나는 경우**

**증상:**

- 개발 서버를 실행하면 "Module not found" 에러 출력
- 또는 "Cannot find module '@/components/...'"
- 페이지를 열어도 하얀 화면만 나오고 아무것도 보이지 않음
- 브라우저 콘솔에 JavaScript 에러 메시지

**원인:**

- Claude가 파일 구조를 잘못 생성했거나 import 경로 오류
- package.json에서 필요한 의존성을 설치하지 않았거나 버전 충돌
- Claude가 생성한 코드가 현재 Next.js 또는 React 버전과 호환되지 않음
- 파일이 만들어지지 않았거나 경로 문제

**해결 방법:**

먼저 에러 메시지를 전체 복사해서 Claude에 전달:

```bash
> 이 에러가 나왔어. 원인 분석하고 수정해줘:

ModuleNotFoundError: Cannot find module '@/components/Question.tsx'
Require stack:
- /users/project/pages/quiz.tsx
- ...
```

또는 브라우저 콘솔 에러를 스크린샷으로 붙여넣기:

```bash
> [브라우저 콘솔 에러 스크린샷 첨부]

브라우저에서 이 에러가 떴어. 뭐가 문제야?
```

Claude가 코드 수정을 제안하면:

```bash
$ npm install
$ rm -rf .next
$ npm run dev
```

만약 특정 패키지 설치 에러가 나면:

```bash
> npm ERR! 이 에러가 나와.
  package.json 다시 생성해주고 npm install하면 될까?
```

**주의사항:**

- "빈 화면"이 나오는 경우는 에러가 콘솔에만 있을 수 있으니 항상 브라우저 콘솔(F12) 확인
- 코드 수정 후 반드시 캐시 삭제 후 재실행: `rm -rf .next && npm run dev`
- 특정 파일만 문제면 해당 파일을 Claude에게 보여주고 "이 파일 전체 다시 만들어줘" 요청 가능

### **상황 2: 코드가 돌아가지만 의도와 다른 결과물이 나오는 경우**

**증상:**

- 페이지는 열리지만 디자인이 생각한 것과 다름
- 버튼 배치가 이상하거나 색상이 다름
- "화면 왼쪽에 사이드바가 있는데 빼고 싶어"
- 모바일에서 레이아웃이 깨짐

**원인:**

- Claude의 초기 이해가 PM의 의도와 약간 다름
- PM이 구체적인 디자인 요구사항을 설명하지 못함 ("깔끔하게" vs "명확한 구체적 스타일")
- 모바일/반응형 디자인 고려 누락

**해결 방법:**

스크린샷을 Ctrl+V로 직접 붙여넣기 (또는 도구로 업로드):

```bash
> [현재 상태의 스크린샷 이미지 붙여넣기]

이건 내가 원한 게 아니야. 수정해줄 사항:
1. 왼쪽 사이드바를 없애줘
2. 카드형 레이아웃으로 바꿔줘
3. 각 카드는 그림자(shadow) 효과 추가
4. 모바일에서도 한 줄에 2-3개 카드가 보이도록
```

더 구체적으로는 참고 자료 제시:

```bash
> [좋아 보이는 웹사이트의 스크린샷]

이 웹사이트의 레이아웃 스타일을 참고해줄 수 있어?
네비게이션, 카드 배치, 색상 톤을 비슷하게.
```

**PM의 작업 흐름:**

- 첫 버전은 거의 항상 "방향 조정"이 필요하다고 생각
- 스크린샷 피드백이 가장 효과적 (말로 설명하는 것보다 보여주는 게 나음)
- 모바일 버전도 꼭 확인 (Ctrl+Shift+M 또는 F12 개발자 도구)

### **상황 3: Claude가 의도치 않게 기존 파일을 수정해서 덮어쓰는 경우**

**증상:**

- package.json이 Claude가 생성한 버전으로 덮어씌워짐
- tailwind.config.js의 커스텀 설정이 사라짐
- 또는 이미 있던 라이브러리 설정이 초기화됨
- "어? 내가 설정한 게 다 없어졌네?"

**원인:**

- Claude가 "이 파일을 완전히 다시 만들겠습니다"라고 판단
- PM이 "package.json을 수정해줘"라고 요청했을 때 전체 파일을 재작성
- 기존 의존성이나 스크립트를 놓침

**해결 방법:**

**방법 1: Plan 모드 활용 (사전 예방)**

Claude Code에서 Plan 모드를 활용:

```bash
> @package.json 이 파일에 "build:tailwind" 스크립트를 추가해줄 수 있어?

# Plan 모드에서 Claude가 제안하는 변경사항을 확인 후 승인
```

Claude가 "파일을 완전히 덮어쓰겠습니다"라고 하면:

- "아니야, 기존 내용은 유지하고 이 부분만 추가해줘"라고 명시

**방법 2: Git으로 복구**

만약 덮어씌워졌으면 git으로 복구:

```bash
$ git status  # 어떤 파일이 변경되었는지 확인

$ git diff package.json  # 변경사항 확인

$ git checkout package.json  # 복구

# 그 후 Claude에게 정확히 추가할 부분만 요청
```

**방법 3: 백업 유지**

중요한 설정 파일은 미리 백업:

```bash
$ cp package.json package.json.backup
$ cp tailwind.config.js tailwind.config.js.backup
```

**PM의 주의사항:**

- "파일 수정" 작업할 때는 항상 변경사항을 먼저 확인하도록 요청
- "이 파일의 [특정 섹션만]" 수정 요청 시 구체적으로 명시
- 프로젝트 초기 단계에는 git으로 모든 변경을 추적하는 것이 필수
