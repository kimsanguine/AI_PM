# 6.2 에이전트 제품 PM의 UI 판단 원칙

## 왜 일반 UI 원칙으로는 부족한가

일반적인 UI 설계는 사용자가 버튼을 누르면 즉시 결과가 나온다는 전제를 깔고 있습니다. 하지만 에이전트는 결과를 예측할 수 없어서 "빈 상태(empty state)"가 아닌 **"진행 상태(running state)"가 기본값**입니다. 에이전트가 무엇을 하고 있는지 사용자에게 보여주지 않으면, 제품이 멈춘 건지 동작 중인 건지 알 수 없어 신뢰가 순식간에 깨집니다.

## Non-negotiable 4원칙

에이전트 UI에는 반드시 포함되어야 할 요소가 있습니다. 아래 4가지는 협상 불가입니다.

| 원칙 | 정의 | 구체 예시 |
|---|---|---|
| **투명성** | 에이전트가 무엇을 하고 있는지 실시간 표시 | "파일 3개 분석 중..." 진행 표시 |
| **사용자 제어** | 언제든 멈추거나 방향 수정 가능 | 취소 버튼, 단계별 승인 |
| **선제 피드백** | 완료 후가 아닌 실행 중 중간 결과 표시 | 초안 스트리밍, 분석 중간 결과 |
| **에러 복구** | 실패 시 이해 가능한 설명 + 다음 행동 안내 | "이 파일을 읽지 못했습니다. 대신 X를 시도할까요?" |

> **PM 판단 기준**: 위 4가지 중 하나라도 빠진 기능은 출시 기준을 통과하지 못한 것입니다. 디자인 완성도보다 이 4원칙을 먼저 점검하세요.

## Agent 특화 KPI

에이전트 제품에서는 일반적인 DAU·전환율보다 **에이전트와 사용자 간의 신뢰 관계**를 측정하는 지표가 더 유의미합니다.

| KPI | 측정 방법 |
|---|---|
| **Task Completion Rate** | 에이전트가 완료를 선언한 작업 중 사용자가 수용한 비율 |
| **Autonomy Trust Score** | 사용자가 중간에 개입(취소·수정)한 비율의 역수 |
| **Error Recovery Rate** | 실패 후 재시도 성공률 |
| **Proactive Action Acceptance** | 에이전트가 먼저 제안한 행동을 사용자가 수락한 비율 |

이 지표들이 낮다면 에이전트의 성능 문제가 아니라 UI의 투명성·제어 부족일 가능성이 높습니다. 먼저 UI를 의심하세요.

## Autonomy-Trust Matrix — 로드맵 우선순위 판단

에이전트 기능의 현재 상태를 아래 Matrix에 배치하면 다음 스프린트 우선순위를 즉시 판단할 수 있습니다.

```
           낮은 자율성          높은 자율성
          ┌─────────────────┬─────────────────┐
높은      │  과보호 구간     │  이상적 구간    │
신뢰도    │  (자율성 확장    │  (현 상태 유지  │
          │   기회)          │   + 기능 확장)  │
          ├─────────────────┼─────────────────┤
낮은      │  MVP 단계        │  위험 구간      │
신뢰도    │  (신뢰 구축      │  (자율성 낮추고 │
          │   우선)          │   신뢰 먼저)    │
          └─────────────────┴─────────────────┘
```

- **고자율 + 고신뢰**: 이상적 상태. 기능을 확장하거나 새로운 에이전트 시나리오를 추가하세요.
- **고자율 + 저신뢰**: 위험 구간. 에이전트가 너무 많이 알아서 행동하고 있습니다. 단계별 승인이나 투명성 UI를 먼저 강화하세요.
- **저자율 + 고신뢰**: 과보호 구간. 사용자가 에이전트를 믿는데 자율성을 제한하고 있습니다. 자율성 확장의 기회입니다.
- **저자율 + 저신뢰**: MVP 단계. 신뢰 구축이 최우선입니다. 기능 추가보다 투명성과 에러 복구를 먼저 완성하세요.

## PM 체크리스트 — 에이전트 기능 출시 전 확인

기능을 출시하기 전에 아래 항목을 직접 확인하세요.

- [ ] 진행 상태가 사용자에게 보이는가
- [ ] 사용자가 언제든 멈출 수 있는가
- [ ] 실패 시 이해할 수 있는 설명이 있는가
- [ ] Task Completion Rate 측정 방법이 정의되었는가

> 4개 중 하나라도 체크하지 못했다면, 그 기능은 아직 출시 준비가 되지 않은 것입니다.
