# 3.2.1 CLAUDE.md 패턴 카탈로그

> 챕터 3.2 (개요·8축) 과 3.2.2 (4.7 시대 작성법) 의 중간 자리. CLAUDE.md 를 **어떤 형태로 배치할 것인가** 의 카탈로그다. 프로젝트의 규모와 팀 구성에 따라 패턴을 고른다.

---

## 3가지 패턴 — 한눈에

| 패턴 | 언제 쓰나 | 예 |
|---|---|---|
| **Mono** | 1인 / 단일 레포 / 단일 도메인 | 본 가이드 레포 (현재) |
| **Multi-folder** | 단일 레포 안에 서브 도메인이 강하게 분리 | monorepo (frontend / backend / docs) |
| **Layered** | 사람 + 팀 + 레포가 다른 룰 가짐 | 다수 개인 + 다수 레포 |

세 패턴은 배타적이지 않다. Layered 의 레포 측은 mono 일 수도 multi-folder 일 수도 있다.

---

## Pattern 1 — Mono CLAUDE.md

**구조**

```
<repo>/
├── CLAUDE.md           # 단일 룰 파일
├── INDEX.md
└── ...
```

**언제**
- 1인 운영, 또는 팀이 작아 컨벤션이 동일.
- 레포 안의 도메인이 균질 (예: 가이드 1권, 단일 앱).
- 8축 룰이 30KB 미만으로 들어감.

**장점**
- 단순. 한 곳만 보면 됨.
- prompt caching 적중률 최고 (변경 점 1개).

**단점**
- 도메인이 갈라지기 시작하면 한 파일이 비대해지고, "어떤 룰이 어느 영역에 적용되는지" 가 흐려진다.
- 200줄을 넘기면 모델이 안 따른다 (4.7 의 effort 낮을 때 특히).

**예시 — 본 가이드 레포**

루트 `/CLAUDE.md` 한 파일로 8축 전부 다룬다. ~90줄, 8KB. 이건 mono 가 맞는 사이즈.

## Pattern 2 — Multi-folder CLAUDE.md

**구조**

```
<repo>/
├── CLAUDE.md                    # 레포 전반 룰 (얇게)
├── frontend/
│   └── CLAUDE.md                # 프런트 전용 룰
├── backend/
│   └── CLAUDE.md                # 백엔드 전용 룰
└── docs/
    └── CLAUDE.md                # 문서 작성 전용 룰
```

Claude Code 는 현재 working directory 의 가장 가까운 CLAUDE.md 부터 위로 거슬러 올라가며 모두 읽는다 (override 또는 누적). 따라서 `frontend/` 안에서 작업하면 root 의 일반 룰 + frontend 의 전용 룰을 둘 다 따른다.

**언제**
- 한 레포에 서브 도메인이 강하게 분리.
- 도메인별 사람·언어·코드 스타일이 다름.
- monorepo 운영.

**장점**
- 도메인별 룰의 명료성.
- 작업 위치에 따라 컨텍스트가 자동으로 좁혀짐.

**단점**
- 룰 중복·충돌의 위험 — One rule = one place 위반 가능.
- 새 폴더가 생길 때마다 CLAUDE.md 신설 여부 결정 필요.

**룰 — 루트 vs 서브폴더**

| 어디에 | 어떤 룰 |
|---|---|
| 루트 | 레포 전반(언어, 모델 ID, PR 표준, 톤) |
| 서브폴더 | 도메인 특화(코드 컨벤션, 도구 선택, 폴더 구조) |

**worked example — monorepo**

```markdown
# <repo>/CLAUDE.md (루트)
- 언어: 한국어 + 영어 용어 원어 유지
- 모델: claude-opus-4-7
- PR 템플릿: .github/PULL_REQUEST_TEMPLATE.md 따름
- 검증 사다리: 변경 후 1단 자동 포맷 필수

# <repo>/frontend/CLAUDE.md
- 프레임워크: Next.js 14 + TypeScript strict
- 스타일: Tailwind, CSS-in-JS 금지
- 테스트: Vitest, RTL
- 4.7 frontend 기본 미감(cream/serif) 피하기 — 명시 색상 사용

# <repo>/backend/CLAUDE.md
- 언어: Python 3.12, ruff format / check
- 프레임워크: FastAPI
- 데이터: Postgres, alembic migrations
- 비밀: .env 사용 금지, 1Password 환경 변수만
```

## Pattern 3 — Layered (Global × Repo × Subfolder)

**구조**

```
~/.claude/
├── CLAUDE.md                   # 사람의 취향 (변하지 않음)
├── guides/                     # lazy reference
└── skills/                     # 전역 스킬

<repo-A>/CLAUDE.md              # 프로젝트 A 규칙
<repo-B>/CLAUDE.md              # 프로젝트 B 규칙
<repo-A>/frontend/CLAUDE.md     # (Pattern 2 결합 가능)
```

**언제**
- 한 사람이 여러 레포에서 작업.
- 사람의 톤·언어·기본 모델은 모든 작업에서 동일.
- 레포마다 도메인·언어·스택이 다름.

**책임 분리**

| 위치 | 무엇을 담는가 | 변경 빈도 | 예 |
|---|---|---|---|
| `~/.claude/CLAUDE.md` | 사람의 취향 (변하지 않음) | 분기 1회 | 톤, 언어, default model |
| `~/.claude/guides/*.md` | 작업 유형별 lazy reference | 월 1회 | writing.md, review.md |
| `~/.claude/skills/*.md` | 반복 워크플로우 | 주 1회 | polish, daily |
| `<repo>/CLAUDE.md` | 프로젝트 규칙 (변함) | 주 1회 | 폴더, 모델 override, scope |
| `<repo>/INDEX.md` | 프로젝트 지식 지도 | 월 1회 | 주석 달린 파일 맵 |

**룰 — 무엇이 어디에 가는가**

- "직설적·단정적 톤" → 전역 (모든 작업에서 동일)
- "이 레포는 한국어만" → 레포 (이 프로젝트에 한정)
- "이 폴더는 TypeScript strict" → 서브폴더 (도메인 특화)

전역과 레포가 충돌하면 — **레포가 이긴다**. 4.7 은 더 가까운(local) 룰을 우선한다. 충돌 자체를 피하는 게 정답이지만.

**worked example — 본 가이드의 layered 적용**

본 레포는 layered 의 절반(`~/.claude/` 스타터를 templates/claude-home/ 에 제공) + 레포는 mono 인 형태:

- `~/.claude/CLAUDE.md` (사용자가 복사) ← `templates/claude-home/CLAUDE.md`
- `<repo>/CLAUDE.md` (본 레포 루트) ← `/CLAUDE.md`
- 서브폴더 CLAUDE.md 없음 (현재 모노 사이즈)

---

## 패턴 선택 가이드 — 5문항

1. 협업자 수가 5명 이상인가? → multi-folder 또는 layered 검토.
2. 한 레포에 도메인이 2개 이상 강하게 분리되어 있는가? → multi-folder.
3. 같은 사람이 여러 레포를 운영하는가? → layered.
4. CLAUDE.md 가 200줄을 넘었는가? → 분리(multi-folder 또는 lazy guide) 검토.
5. 룰의 변경 빈도가 위치별로 다른가? (사람 룰은 안 변함, 프로젝트 룰은 자주 변함) → layered.

3개 이상 yes → mono 에서 벗어날 시점.

---

## 안티패턴

- **모든 CLAUDE.md 가 동일한 내용** — 그러면 layered 가 아니라 그냥 중복. 한 곳만 남기고 삭제.
- **루트 CLAUDE.md 가 1KB 이하** — 너무 비어 있음. 톤·모델 ID·검증 사다리 1단 정도는 박아라.
- **서브폴더 CLAUDE.md 가 100KB** — 그건 매뉴얼이지 CLAUDE.md 가 아님. lazy guide 로 분리.
- **`~/.claude/CLAUDE.md` 와 `<repo>/CLAUDE.md` 가 같은 룰 — 한쪽 삭제**.
- **CLAUDE.md 안에 시크릿** — 절대 금지. `.gitignore` 가 막아주지 않음 (글로벌 ~/.claude/ 는 git 밖이지만, 레포 CLAUDE.md 는 커밋됨).

---

## 운영 권장

- mono → multi-folder 전환 시점: 한 영역의 룰이 30줄 이상 되면 그 폴더로 옮겨라.
- multi-folder → layered 전환 시점: 두 번째 레포가 생기면 사람의 취향을 `~/.claude/` 로 올려라.
- 패턴 결정 후, `docs/operations/closed-loop.md` 의 폐루프(/retro) 에서 패턴 자체도 점검해라 — 분기 1회.
