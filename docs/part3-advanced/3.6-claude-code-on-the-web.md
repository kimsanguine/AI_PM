# 3.6 Claude Code on the web — 웹·모바일·GitHub Action 실행 환경

> 챕터 1.3 (설치와 첫 실행) 은 로컬 CLI 가 출발점이었다. 본 챕터는 **로컬 외부에서 Claude Code 를 돌리는 4가지 환경** — 웹, 모바일, GitHub Action, 그리고 클라우드 워크트리(remote container) 다. 핵심 차이는 "**누가 컨테이너를 관리하고, 네트워크가 어디까지 열려 있는가**" 다.

---

## 4가지 실행 표면

| 환경 | 누가 트리거 | 컨테이너 | 네트워크 | 주 용도 |
|---|---|---|---|---|
| **CLI (로컬)** | 사람 | 로컬 머신 | 사용자 환경 | 일상 작업, 코드 편집 |
| **Web/Mobile** | 사람 (브라우저·앱) | Anthropic 관리 ephemeral 컨테이너 | 정책 가능 (open / closed) | 출장 중, 가벼운 작업 |
| **GitHub Action** | git push / PR / 댓글 | runner | repo + GitHub API | PR 리뷰, CI 게이트 |
| **Cloud worktree** | 사람 또는 API | Anthropic ephemeral | 정책 가능 | 병렬 세션 격리, 자동화 |

본 가이드의 v1.1 작업 자체가 cloud worktree 위에서 진행되었다 — 사용자 머신이 아니라 격리된 컨테이너에서 코드 편집·git push 가 일어나는 방식.

## 무엇이 달라지는가 — 로컬 CLI 와 비교

- **컨테이너 ephemeral** — 세션 끝나면 환경이 사라진다. **커밋·푸시되지 않은 변경은 잃는다**.
- **네트워크 정책** — 사용자가 선택한 정책에 따라 외부 호출이 제한될 수 있다 (사내 API / 공개 도메인 only 등).
- **설정 인프라** — 로컬 `~/.claude/` 가 없다. 환경 변수와 setup script 로 대신.
- **GitHub MCP 만 접근** — `gh` CLI 가 없고, github MCP 툴로만 PR/issue 작동.

이 차이가 운영 룰을 바꾼다 — **항상 commit → push** 가 기본 동작이 된다. 미커밋 변경은 컨테이너 회수와 함께 사라진다.

## 검증 사다리 3단 (시각 렌더) 의 새 가능성

챕터 2.6 의 검증 사다리는 4단계 — 자동 포맷 → 테스트 → **시각** → LLM 리뷰. 3단(시각) 은 로컬 CLI 에서는 사람이 브라우저로 직접 봐야 했다. Web 환경에서는 다르다:

- 모델이 직접 브라우저(예: Chrome MCP, Playwright MCP) 를 띄워 페이지를 캡처·검수할 수 있다.
- 디자인 변경 PR 의 미리보기 스크린샷을 모델이 첨부 가능.
- 4.7 의 고해상도 이미지 지원(2576px) 과 결합 → 픽셀 단위 검수 가능.

**worked example**

```
PM: "이 PR 의 로그인 페이지가 mobile 에서 한국어가 잘리는지 봐줘."
Claude (web 환경):
  1. PR preview URL 띄움
  2. 모바일 viewport 로 캡처
  3. 한국어 텍스트 영역 bounding box 추출
  4. 잘림 여부 보고 + 스크린샷 첨부
```

→ 시각 검증 3단이 자동화된다. 사람은 4단(판단) 만 하면 된다.

## GitHub Action 통합

GitHub Action 으로 Claude Code 를 발화시키는 경우 — PR 댓글, 라벨, 일정 — 운영 룰:

- **scope 제한**: 단일 PR / 단일 레포에 한정. 다른 레포 cross-edit 금지 (본 가이드 CLAUDE.md 7축 룰).
- **확인 후 행동**: 머지·force-push 같은 위험 행동은 사람의 명시적 댓글이 있을 때만.
- **PR 활동 구독**: PR 댓글·CI 이벤트를 자동으로 받아 다음 행동(재시도, 수정 푸시) 결정.

본 가이드의 `.github/workflows/` 는 lint / links / frontmatter 3종이지만, Claude Code 자체를 호출하는 워크플로우는 v1.3 의 자동화 절에서 다룬다.

## 네트워크 정책 — 3가지 모드

| 모드 | 무엇이 열림 | 언제 |
|---|---|---|
| **None** | 외부 호출 전부 차단 | 코드 편집·문서 작성, 비밀 보호 |
| **Curated** | 허용 도메인 리스트 (npm registry, GitHub, Anthropic) | 의존성 설치 + 일반 작업 |
| **Open** | 임의 호출 허용 | 자료 수집·웹 검색 필요 |

본 가이드 작업처럼 markdown 만 다루면 **None 또는 Curated** 가 적합. **Open** 은 의도된 검색·외부 자료 fetch 때만 켠다. SoT 정책 (`docs/operations/sot-policy.md`) 의 promotion (Notion → GitHub) 같은 자동화는 Curated 로 운영.

## Cloud worktree — 격리된 병렬 세션

`isolation: worktree` 옵션은 한 세션이 별도 워크트리(임시 브랜치 + 작업 디렉토리) 에서 작업하게 한다.

- 같은 레포에서 **3–6 병렬 세션** (챕터 2.5) 을 안전하게.
- 각 세션은 격리된 디렉토리 → 충돌 없음.
- 변경 없으면 자동 정리, 있으면 path + branch 반환.

**worked example**

```
세션 A (worktree-A) — README 인코딩 수정
세션 B (worktree-B) — 챕터 2.2 4.7 재검수
세션 C (worktree-C) — 신규 스킬 작성
```

세 세션이 같은 레포의 다른 브랜치에서 동시에 진행된다. PM 은 Stop 훅(2.7) 알림으로 어느 세션이 끝났는지 안다.

## 안티패턴

- **로컬 CLI 룰을 그대로 web 으로 가져옴** — `~/.claude/` 없는 환경에서 lazy guide / 전역 skill 이 작동 안 함. web 용 셋업 스크립트로 주입 필요.
- **세션 안에서 안 끝낸 작업** — commit/push 안 하면 잃는다. PR 활동 구독 권장.
- **Open 네트워크 정책 default** — 위험. 의도된 검색에만 일시적으로.
- **GitHub Action 에서 머지 자동 승인** — 사람의 확인 없는 머지는 금지. 4.7 의 literal following 으로도 막을 수 없다.

## 빠른 진단 — 어느 표면을 쓸까

| 상황 | 권장 |
|---|---|
| 일상 코드 편집·문서 작업 | CLI (로컬) |
| 출장·이동 중 빠른 review | Web / Mobile |
| PR 자동 review·CI 자동 수정 | GitHub Action |
| 병렬 세션 운영 / 자동화 | Cloud worktree |
| 디자인 변경 시각 검증 | Web (브라우저 MCP) |

## 마무리

Claude Code 는 단일 CLI 가 아니라 **여러 실행 표면을 가진 에이전트 시스템** 이다. 표면이 늘어날수록 "어디서 무엇을 어떤 룰로 돌릴 것인가" 의 결정이 PM 의 새 책임이 된다. 가이드의 8축 CLAUDE.md (특히 7축 Out-of-scope) 와 SoT 정책이 이 결정의 골격이다.
