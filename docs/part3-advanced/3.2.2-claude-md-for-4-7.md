# 3.2.2 4.7 시대의 CLAUDE.md

> Claude Opus 4.7 부터 모델은 지시를 **literal 하게** 따른다. 모호한 룰은 무시되고, 명시되지 않은 범위는 generalize 되지 않는다. CLAUDE.md 의 작성법은 이 변화 위에서 다시 설계되어야 한다.

---

## 왜 4.7 에서 CLAUDE.md 가 더 중요해졌나

Claude Opus 4.7 의 행동 변화 중 CLAUDE.md 설계와 직결되는 4가지:

| 변화 | CLAUDE.md 에 미치는 영향 |
|---|---|
| **More literal instruction following** | 모호한 룰("적절히") 은 무시. 모든 룰에 측정 가능한 기준 필요. |
| **Adaptive thinking off-by-default** | 어떤 작업에 think 를 켤지 명시하지 않으면 항상 꺼짐. |
| **Fewer subagents / tool calls by default** | fan-out 트리거를 CLAUDE.md 에 박지 않으면 단일 세션화. |
| **Response length calibrates to perceived complexity** | "짧게" 가 아니라 "800–1200자" 같은 숫자가 필요. |

핵심: 4.7 은 친절하게 추측해주지 않는다. **CLAUDE.md 가 명시하지 않은 것은 모델이 채워주지 않는다.**

## 8축 프레임워크

기존 5축(Project / Conventions / Workflows / Tools / Voice) 에 3개를 추가해 **8축** 으로 확장한다. 추가 축은 모두 4.7 의 literal following 에 대응하기 위한 것이다.

| # | 축 | 무엇을 담는가 | 4.7 대응 |
|---|---|---|---|
| 1 | Project | 정체성 — 누가, 무엇을, 누구를 위해 | (기본) |
| 2 | Conventions | 글쓰기·코딩 컨벤션 | (기본) |
| 3 | Workflows | 슬래시·스킬에서 따를 Task/Scope/Length/Format | 길이·포맷 명시 |
| 4 | Tools | 모델 ID, effort, MCP 등 | xhigh/high, claude-opus-4-7 명시 |
| 5 | Voice | 톤·언어·금지어 | direct 톤 명시 |
| 6 | **Examples** | `<example>` 태그 few-shot 3–5개 | few-shot 으로 모호함 제거 |
| 7 | **Escalation / Out-of-scope** | 일반화 금지, 본 레포 밖 금지 | literal following 대응 |
| 8 | **Knowledge Map** | INDEX.md 진입점, 검증 사다리, 폐루프 | 매번 시키지 않음 |

이 가이드의 루트 `CLAUDE.md` 가 8축의 worked example 이다. 본 챕터 끝에서 전문 인용한다.

## 4.7-tuned 작성 룰 10개 (모두 긍정형)

부정형("don't…") 대신 긍정형("do X instead") 으로 써라 — 4.7 은 부정형을 ambiguous 하게 처리한다.

1. **모든 워크플로우 항목은 4섹션 — Task / Scope / Length / Format.**
2. **모든 예시는 `<example>` 태그로 3–5개**. good / bad 짝으로 대비.
3. **일반화는 명시적으로 — "every chapter" / "every workflow" / "본 레포 안의 모든 .md"** 라고 쓴다. "전반적으로" 는 무시된다.
4. **thinking 트리거 명시** — "전략·아키텍처·trade-off 비교일 때만 think". 그 외는 off.
5. **모델 ID 는 `claude-opus-4-7` 정자.** 마케팅 이름 ("Claude 4 Opus") 금지.
6. **출력 길이는 숫자로** — "짧게" 대신 "600–1,200자".
7. **출력 포맷 양자택일** — prose / bullet / table / JSON 하나로. 혼용 금지.
8. **부정형은 긍정형으로 변환** — "don't use ellipsis" → "use periods to end sentences".
9. **Long doc 패턴** — `<documents>` 안 `<document>` + `<source>` 태그 + quote-grounding.
10. **변경 시 last_updated + CHANGELOG.md 갱신** — 시간 정보가 없으면 stale 판단 불가.

## Files-as-knowledge-base + Prompt Caching

4.7 의 1M 컨텍스트 + prompt caching 을 잘 쓰는 구조:

```
┌─────────────────────────────────────────┐
│  안정 영역 (자주 안 바뀜)               │  ← cache hit
│  - ~/.claude/CLAUDE.md (전역 취향)      │
│  - <repo>/CLAUDE.md (프로젝트 규칙)     │
│  - <repo>/INDEX.md (지식 지도)          │
├─────────────────────────────────────────┤
│  세션별 휘발성 (매번 다름)              │  ← cache miss
│  - 현재 작업 파일들                     │
│  - 사용자 프롬프트                      │
└─────────────────────────────────────────┘
```

**원칙**: 안정 영역을 **위에**, 휘발성을 **아래**에. 안정 영역이 자주 바뀌면 캐시가 무력화돼 비용이 폭증한다. 따라서 **CLAUDE.md 의 변경 빈도는 의도적으로 낮춰라**. 잦은 변경이 필요하면 그건 CLAUDE.md 가 아니라 lazy-load guide (`~/.claude/guides/*.md`) 의 영역이다.

## 전역 ↔ 레포 책임 분리

| 위치 | 무엇을 담는가 | 변경 빈도 | 예 |
|---|---|---|---|
| `~/.claude/CLAUDE.md` | 사람의 취향 (변하지 않음) | 분기 1회 | 톤, 언어, 직설성 |
| `~/.claude/guides/*.md` | 작업 유형별 lazy reference | 월 1회 | writing.md, review.md, evals.md |
| `~/.claude/skills/*.md` | 반복 워크플로우 | 주 1회 | polish, daily |
| `<repo>/CLAUDE.md` | 프로젝트 규칙 (변함) | 주 1회 | 폴더, 모델, scope |
| `<repo>/INDEX.md` | 프로젝트 지식 지도 | 월 1회 | 주석 달린 파일 맵 |
| `<repo>/.claude/agents/*.md` | 프로젝트 전용 서브에이전트 | 분기 1회 | engineering-reviewer 등 |

룰: **같은 룰이 전역과 레포 양쪽에 있으면 한쪽 삭제**. One rule = one place.

## Worked example — 본 가이드의 루트 CLAUDE.md (발췌)

```markdown
## 1. Project
- PM 을 위한 Claude Code 실전 가이드. 1차 독자: 한국어 PM (J/P/L).

## 2. Conventions
- 한국어 본문 + 영어 용어 원어 유지.
- 챕터 본문 분량 8–25KB.

## 3. Workflows — 모든 슬래시/스킬에 4섹션
- Task — 무엇을 한다.
- Scope — 어디까지, 어디는 안 한다.
- Length — 800–1,200자 같은 숫자.
- Format — markdown / JSON / XML 중 하나.

## 4. Tools
- 모델: claude-opus-4-7
- effort: xhigh (편집) · high (리뷰) · medium (단순 lookup)
- thinking: adaptive(off-by-default), 다단계 추론 시에만 on

## 5. Voice
- 직설적·단정적. 결론을 첫 문장에.
- 사용자의 framing 에 무조건 동의하지 않는다.

## 6. Examples
- 모든 챕터·스킬·커맨드는 3–5개의 <example> 블록.
- good / bad 짝.

## 7. Escalation / Out-of-scope
- "Apply to every chapter" 같은 명시 scope 없으면 한 챕터에만 적용.
- 본 레포 밖 작업 금지.
- subagent fan-out 기준: 3개 챕터 동시 검수 또는 6개 파일 동시 grep.

## 8. Knowledge Map
- 작업 전 INDEX.md 를 먼저 읽고 관련 파일만 펼친다.
- 변경 시 CHANGELOG.md + last_updated 갱신.
```

이 8축은 "이 레포가 어떻게 자기 자신을 운영하는지" 의 실증이다. 가이드는 가르치고, 레포는 그대로 한다.

## 마이그레이션 체크리스트 — 4.6 → 4.7 CLAUDE.md

기존 CLAUDE.md 를 4.7 로 옮길 때 점검할 항목:

- [ ] `extended thinking budget` 같은 4.6 표현 제거. `adaptive thinking` 으로 통일.
- [ ] 모델 ID 를 `claude-opus-4-7` 로 정자화.
- [ ] effort 권장값 (xhigh/high/medium/low) 명시.
- [ ] "subagent 를 적극 활용" 같은 가정 제거 — 4.7 은 기본적으로 덜 띄움. fan-out 트리거 명시.
- [ ] "이모지·warm tone" 가정 제거 — 4.7 은 더 direct. 필요하면 명시.
- [ ] 출력 길이를 숫자로 표기 (예: 600–1,200자).
- [ ] 부정형 룰("don't…") 을 긍정형("use … instead")으로 변환.
- [ ] few-shot 예시를 `<example>` 태그로 감싸기.
- [ ] last_updated · claude_model · claude_code_version 메타 추가.
- [ ] 토크나이저 변경 반영: max_tokens 헤드룸 1.0~1.35× 확보 (CLAUDE.md 직접 영향은 작지만, 호출 코드 갱신).

## 안티패턴

- **무의미 형용사 룰** — "적절히", "효과적으로", "다양한" 같은 단어는 4.7 이 따르지 않는다.
- **부정의 산** — "X 하지 마라" 가 10개 이상 — 모델이 따르지 못한다. 긍정형 3–5개로 압축.
- **모든 워크플로우에 'think on'** — latency 와 비용을 망친다. think 는 다단계 추론 시에만.
- **레포 CLAUDE.md 가 100KB** — caching 적중률이 무너지고 모델 컨텍스트의 절반이 룰에 잡힌다. 30KB 미만 권장.
- **One rule = many places** — 같은 룰이 CLAUDE.md, guides, skills, hooks 4곳에 — 충돌 폭탄.

## 빠른 진단 — 내 CLAUDE.md 가 4.7-tuned 인가

체크 5개. 3개 이하면 재작성 권장.

1. 모든 룰에 측정 가능한 기준(숫자·열거·정자) 이 있는가?
2. 모든 예시가 `<example>` 태그 안에 들어 있는가?
3. fan-out 트리거가 명시되어 있는가? (몇 개 파일 / 몇 개 챕터)
4. thinking on / off 의 트리거 조건이 명시되어 있는가?
5. last_updated · claude_model 메타가 있는가?

## 마무리

4.7 시대 CLAUDE.md 는 "친절한 안내문" 이 아니라 "정확한 사양서" 다. 모호하게 쓰면 모델은 정확히 그 모호함만큼만 따른다. 본 레포의 루트 CLAUDE.md (전문은 `/CLAUDE.md`) 를 출발점으로 자신의 프로젝트로 가져가라 — 그리고 1주 후 `/retro` (챕터 3.3) 로 자주 고치는 항목을 다시 흡수해라. 폐루프가 작동하면, CLAUDE.md 는 살아있는 룰북이 된다.
