# 3.7 플러그인과 마켓플레이스 — 팀 워크플로를 한 번에 설치하기

> **핵심 개념**: 지금까지는 슬래시 명령([3.3](3.3-slash-commands.md)), 스킬([3.4](3.4-custom-skills.md)), 훅([../part2-basics/2.7-hooks.md](../part2-basics/2.7-hooks.md))을 **하나씩** 만들어 썼습니다. 플러그인은 이 조각들을 **하나의 묶음**으로 포장해, 명령 한 줄로 설치·공유할 수 있게 합니다. 마켓플레이스는 그런 플러그인을 모아 놓은 **카탈로그**입니다. PM 팀이 같은 워크플로를 똑같이 쓰도록 표준화하는 가장 빠른 길입니다.

---

## 왜 PM에게 플러그인이 필요한가

스킬을 하나 잘 만들었다고 해 봅시다. PRD 초안을 소크라틱 질문으로 뽑아 주는 스킬입니다. 나 혼자 쓰면 좋습니다. 그런데 팀원 다섯 명이 같은 스킬을 쓰게 하려면 어떻게 해야 할까요.

스킬 폴더를 압축해 메신저로 보내고, "이걸 `~/.claude/skills/` 밑에 풀어 넣으세요"라고 안내하고, 누군가는 경로를 틀리고, 누군가는 버전이 달라지고 — 이런 과정이 반복됩니다. 도구 지도([../part2-basics/2.0-toolkit-map.md](../part2-basics/2.0-toolkit-map.md))에서 본 다섯 범주(명령·스킬·서브에이전트·훅·MCP)를 **각각 따로 배포하면** 관리 비용이 곱으로 늘어납니다.

플러그인은 이 문제를 정면으로 풉니다. 여러 구성요소를 한 폴더에 담고, 팀원은 명령 두 줄로 똑같이 설치합니다.

> **공식 정의** (code.claude.com/docs/en/discover-plugins): "플러그인은 스킬, 에이전트, 훅, MCP 서버로 Claude Code를 확장합니다(Plugins extend Claude Code with skills, agents, hooks, and MCP servers)."

즉 플러그인 하나에 다음을 함께 담을 수 있습니다.

| 구성요소 | 무엇인가 | 이 책의 챕터 |
| :--- | :--- | :--- |
| **명령(commands)** | `/이름` 형태의 슬래시 명령 | [3.3 슬래시 명령](3.3-slash-commands.md) |
| **스킬(skills)** | `SKILL.md`로 정의한 멀티스텝 워크플로 | [3.4 커스텀 스킬](3.4-custom-skills.md) |
| **에이전트(agents)** | 특정 작업 전용 서브에이전트 | [2.4 커스텀 서브에이전트](../part2-basics/2.4-custom-subagents.md) |
| **훅(hooks)** | 특정 시점에 자동 실행되는 동작 | [2.7 훅](../part2-basics/2.7-hooks.md) |
| **MCP 서버** | 외부 도구·데이터 연결 | [3.1 MCP 통합](3.1-mcp-integration.md) |

## 마켓플레이스 동작 원리 — 두 단계

마켓플레이스를 쓰는 과정은 **두 단계**입니다. 공식 문서는 이를 앱 스토어에 비유합니다.

> "마켓플레이스를 쓰는 것은 두 단계 과정입니다: 마켓플레이스 추가 → 개별 플러그인 설치. 스토어를 추가하면 컬렉션을 둘러볼 수 있게 되지만, 어떤 앱을 받을지는 여전히 직접 고릅니다(adding the store gives you access to browse its collection, but you still choose which apps to download individually)." — code.claude.com/docs/en/discover-plugins

### 1단계 — 마켓플레이스 추가

카탈로그를 Claude Code에 **등록**합니다. 이 시점에는 아무것도 설치되지 않습니다. 둘러볼 목록만 생깁니다.

GitHub 저장소는 `owner/repo` 형식으로 추가합니다. 공식 문서 표현 그대로, `owner`는 GitHub 사용자명 또는 조직명이고 `repo`는 저장소 이름입니다.

```shell
/plugin marketplace add anthropics/claude-code
```

> `/plugin marketplace` 대신 `/plugin market`을, `remove` 대신 `rm`을 줄여 쓸 수 있습니다(code.claude.com/docs/en/discover-plugins).

### 2단계 — 개별 플러그인 설치

추가한 마켓플레이스에서 원하는 플러그인을 고릅니다. 설치 명령의 형식은 `플러그인이름@마켓플레이스이름`입니다.

```shell
/plugin install 플러그인이름@마켓플레이스이름
```

설치 후에는 `/reload-plugins`를 실행해 활성화합니다. 공식 문서에 따르면 **플러그인 스킬은 플러그인 이름으로 네임스페이스가 붙습니다**. 예를 들어 `commit-commands` 플러그인은 `/commit-commands:commit` 같은 형태로 스킬을 제공합니다(code.claude.com/docs/en/discover-plugins).

### 둘러보기 — `/plugin` 화면

명령을 외우지 않아도 됩니다. `/plugin`만 입력하면 탭으로 나뉜 관리 화면이 열립니다. 공식 문서는 네 개 탭을 이렇게 설명합니다(code.claude.com/docs/en/discover-plugins).

- **Discover**: 추가한 마켓플레이스의 플러그인 둘러보기
- **Installed**: 설치한 플러그인 보기·관리
- **Marketplaces**: 마켓플레이스 추가·삭제·업데이트
- **Errors**: 플러그인 로딩 오류 확인

플러그인을 고르면 **설치 전에 무엇이 추가되는지** 미리 보여 줍니다. 명령·에이전트·스킬·훅·MCP 서버 목록과, 매 턴 컨텍스트에 더해지는 토큰 비용 추정치까지 표시됩니다(Claude Code 버전에 따라 항목이 다를 수 있음, code.claude.com/docs/en/discover-plugins).

## 설치 범위(scope) — 나만 vs 팀 전체

플러그인을 설치할 때 **누구에게 적용할지**를 고릅니다. 공식 문서가 정의하는 범위는 세 가지입니다(code.claude.com/docs/en/discover-plugins).

| 범위 | 적용 대상 | PM 활용 |
| :--- | :--- | :--- |
| **User scope**(기본값) | 내 모든 프로젝트 | 개인 워크플로 |
| **Project scope** | 이 저장소의 모든 협업자 | `.claude/settings.json`에 기록되어 **팀 공유** |
| **Local scope** | 이 저장소에서 나만 | 실험·임시 사용 |

`/plugin install`을 그냥 실행하면 User scope로 설치됩니다. 다른 범위를 고르려면 `/plugin`을 열고 Discover 탭에서 플러그인에 **Enter**를 눌러 범위를 선택합니다.

> ⚠️ **신뢰할 수 있는 출처만 설치하세요.** 공식 문서는 경고합니다: "플러그인은 사용자 권한으로 임의의 코드를 실행할 수 있는, 신뢰도가 높은 구성요소입니다(highly trusted components that can execute arbitrary code on your machine)." Anthropic은 플러그인에 포함된 MCP 서버·파일·소프트웨어를 통제하거나 검증하지 않습니다(code.claude.com/docs/en/discover-plugins).

## PM 팀 표준화 — 모두가 같은 워크플로를 쓰게 하기

여기가 PM에게 가장 중요한 부분입니다. 팀원이 저장소를 열면 **자동으로 같은 플러그인 설치를 권유받게** 만들 수 있습니다. 수동 안내문도, 압축 파일도 필요 없습니다.

공식 문서의 "팀 마켓플레이스 구성(Configure team marketplaces)" 절차에 따르면, 프로젝트의 `.claude/settings.json`에 `extraKnownMarketplaces`를 적습니다. 팀원이 그 저장소 폴더를 신뢰(trust)하면 Claude Code가 마켓플레이스와 플러그인 설치를 권유합니다(code.claude.com/docs/en/discover-plugins).

```json
{
  "extraKnownMarketplaces": {
    "my-team-tools": {
      "source": {
        "source": "github",
        "repo": "your-org/claude-plugins"
      }
    }
  }
}
```

이렇게 하면 신입 PM이 들어와도 "PRD 스킬은 어디서 받나요?" 같은 질문이 사라집니다. 저장소를 여는 순간 표준 도구 세트가 따라옵니다.

## 실전 예시 — 이 책의 자료를 플러그인으로 설치하기

이 책의 저장소 자체가 하나의 마켓플레이스로 구성되어 있습니다(`.claude-plugin/marketplace.json`). 여기에는 PM 핵심 워크플로를 묶은 `ai-pm` 플러그인이 정의되어 있습니다.

설치는 위에서 본 두 단계 그대로입니다.

```shell
/plugin marketplace add kimsanguine/AI_PM
/plugin install ai-pm@ai-pm
```

> 설치 형식이 `플러그인이름@마켓플레이스이름`이므로, 플러그인과 마켓플레이스 이름이 둘 다 `ai-pm`인 이 경우엔 `ai-pm@ai-pm`이 됩니다.

설치 후 `/reload-plugins`로 활성화하면, 이 책에서 만든 PM 스킬들이 한 번에 들어옵니다.

- `prd-generator` — 소크라틱 질문 기반 PRD 초안 생성
- `discovery-synthesizer` — 디스커버리 인터뷰 결과 종합
- `competitor-battlecard` — 경쟁사 배틀카드 작성
- `experiment-analyzer` — 실험·A/B 결과 분석
- `kpi-card-builder` — KPI 카드 구성

여기에 `templates/commands/`에 담긴 슬래시 명령(예: `/prd`, `/discovery`, `/competitor`)도 함께 설치됩니다.

> **네임스페이스 기억하기**: 플러그인으로 설치한 스킬은 플러그인 이름이 앞에 붙습니다. 즉 `ai-pm` 플러그인의 PRD 스킬은 `/ai-pm:prd-generator` 형태로 호출합니다.

## 외부 플러그인 설치 보안 체크

외부에서 플러그인이나 스킬을 받기 전에 아래 4단계를 확인하세요.

1. **SKILL.md 직접 읽기** — 어떤 명령어와 권한을 요청하는지 확인
2. **번들 스크립트 확인** — `scripts/` 디렉토리에 외부 네트워크 호출이나 파일 삭제 여부
3. **제작자 신뢰도 확인** — GitHub 프로필, 스타 수, 커뮤니티 평판
4. **처음엔 ask 모드** — `--permission-mode ask`로 테스트 후 전환

> 보안 연구(ToxicSkills)에서 공개 스킬의 상당수에 prompt injection 취약점이 발견됐습니다. 신뢰할 수 없는 출처의 스킬은 주의가 필요합니다.

## AI_PM vs 다른 플러그인 생태계

| | AI_PM 플러그인 | Anthropic `knowledge-work-plugins` |
|---|---|---|
| 특화 영역 | PM 워크플로 (Discovery·기획·전달) | 일반 지식 작업 (문서 요약·회의 정리) |
| 대상 | 제품 관리자 | 지식 노동자 전반 |
| 사용 방식 | 팀 CLAUDE.md에 표준 등록 | 프로젝트별 필요에 따라 추가 |

두 가지를 함께 사용하면 PM 업무 전반을 커버할 수 있습니다.

## 정리

| 질문 | 답 |
| :--- | :--- |
| 플러그인이 무엇을 묶나 | 명령·스킬·에이전트·훅·MCP 서버를 한 폴더에 |
| 어떻게 받나 | `/plugin marketplace add` → `/plugin install` 두 단계 |
| 팀에 어떻게 표준화하나 | 프로젝트 `.claude/settings.json`의 `extraKnownMarketplaces` |
| 활성화는 | 설치 후 `/reload-plugins` |
| 주의할 점 | 신뢰할 수 있는 출처만 설치 |

플러그인은 [3.3 슬래시 명령](3.3-slash-commands.md)·[3.4 커스텀 스킬](3.4-custom-skills.md)·[2.7 훅](../part2-basics/2.7-hooks.md)에서 만든 것을 **버리지 않고**, 그 위에 "배포·공유"라는 한 겹을 더합니다. 혼자 쓰던 워크플로를 팀의 표준으로 끌어올리는 단계입니다.

> 본 챕터의 명령 형식과 동작은 code.claude.com/docs/en/discover-plugins 및 code.claude.com/docs/en/plugin-marketplaces 문서를 기준으로 작성했습니다(2026-06-02 확인).
