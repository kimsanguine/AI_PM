# 2.0 Claude Code 도구 지도 — 5가지 범주

> Part 2 에 들어가기 전에 먼저 전체 지형을 한눈에 본다. Claude Code 에는 기억·스킬·자동 실행·대리인·확장이라는 다섯 갈래의 도구가 있다. 각각을 깊이 파기 전에, "이게 어느 갈래에 속하는가" 를 분류하는 멘탈모델부터 세우는 것이 목적이다.

---

## 왜 지도가 먼저인가

비개발자 PM 이 Claude Code 를 처음 접하면 흔히 빠지는 함정이 있다 — **새로 들은 기능을 닥치는 대로 다 설치하는 것**. hooks 글을 읽으면 hook 을, MCP 소개를 보면 MCP 를, 스킬 영상을 보면 스킬을 한꺼번에 깔아두고는 정작 무엇이 무엇인지 구분하지 못한다.

해법은 도구를 **다섯 범주로 분류하는 사고** 다. 새 기능이 나오면 먼저 "이건 다섯 중 어디에 속하지?" 를 묻는다. 그러면 "또 하나 깔아야 하나" 가 아니라 "내가 이미 쓰는 범주의 변형인가, 정말 새 범주인가" 를 판단할 수 있다. 분류가 되면 과잉 설치는 저절로 줄어든다.

## 5가지 범주

| # | 범주 | 한 줄 정의 | 대표 형태 |
|---|---|---|---|
| 1 | **기억** | 매 세션 모델이 자동으로 읽는 영구 메모 | `CLAUDE.md`, 프로젝트 메모리 |
| 2 | **스킬** | 호출하면 펼쳐지는 재사용 작업 묶음 | custom skill |
| 3 | **자동 실행** | 특정 시점에 시스템이 알아서 돌리는 스크립트 | hooks |
| 4 | **대리인** | 별도 맥락에서 일을 위임받아 처리하는 하위 일꾼 | subagent, agent team |
| 5 | **확장** | 외부 시스템·기능을 Claude Code 에 연결 | MCP, plugin |

### 1) 기억 — 모델이 매번 자동으로 읽는 것

세션을 새로 열 때마다 다시 설명하지 않아도 되도록, 프로젝트의 규칙·맥락·용어를 파일에 적어 두는 범주다. 모델이 **부르지 않아도 스스로** 읽는다는 점이 핵심이다.

→ 심화: [2.3 프로젝트 메모리](2.3-project-memory.md) · [3.2 CLAUDE.md 심화](../part3-advanced/3.2-claude-md-deep-dive.md)

### 2) 스킬 — 부르면 펼쳐지는 작업 묶음

자주 반복하는 일련의 절차를 하나의 이름으로 묶어 두고, 필요할 때 호출해 쓰는 범주다. 기억이 "항상 켜져 있는" 것이라면, 스킬은 **부를 때만 펼쳐지는** 것이다.

→ 심화: [3.4 커스텀 스킬](../part3-advanced/3.4-custom-skills.md)

### 3) 자동 실행 — 시점이 되면 알아서 도는 것

편집 직후, 세션 시작 시처럼 **정해진 시점** 에 사람의 지시 없이 시스템이 스크립트를 돌리는 범주다. 모델의 추론이 아니라 harness 가 실행한다는 점에서 스킬·대리인과 구분된다.

→ 심화: [2.7 Hooks](2.7-hooks.md)

### 4) 대리인 — 일을 위임받는 하위 일꾼

본 세션과는 **별도의 맥락** 에서 특정 임무를 받아 처리하고 결과만 돌려주는 범주다. 하나에게 맡기면 서브에이전트, 여럿을 동시에 돌리면 에이전트 팀이다.

→ 심화: [2.4 커스텀 서브에이전트](2.4-custom-subagents.md) · [2.5 에이전트 팀](2.5-agent-teams.md)

### 5) 확장 — 외부와 연결

Claude Code 자체에는 없는 기능·데이터(외부 API, 데이터베이스, 도구 묶음)를 끌어와 붙이는 범주다. 단건 연결은 MCP, 여러 기능을 한 꾸러미로 배포하면 plugin 이다.

→ 심화: [3.1 MCP 연동](../part3-advanced/3.1-mcp-integration.md) · [3.7 플러그인과 마켓플레이스](../part3-advanced/3.7-plugins-and-marketplace.md)

## "어느 범주인가" 로 분류하는 연습

새 기능을 마주쳤을 때 아래 순서로 물어본다.

1. **부르지 않아도 항상 적용되나?** → 그렇다면 **기억**.
2. **내가 호출할 때만 펼쳐지나?** → **스킬**.
3. **정해진 시점에 시스템이 알아서 돌리나?** → **자동 실행**.
4. **별도 맥락에서 일을 위임하나?** → **대리인**.
5. **외부 시스템·기능을 끌어오나?** → **확장**.

다섯 중 어디에도 안 맞으면, 그때 비로소 "정말 새로운 무언가" 일 수 있다. 대부분은 다섯 중 하나의 변형이다. 이 분류 습관 하나가 "도구 한꺼번에 깔기" 를 막아 준다.

## 다섯 범주는 서로 보완한다

각 범주는 독립이 아니라 맞물려 돈다. 예를 들어 **자동 실행**(hook)이 세션 시작 시 **기억**(CLAUDE.md)을 주입하고, **대리인**(서브에이전트)이 검토를 마치면 다시 **자동 실행**(`SubagentStop`)이 알림을 띄운다. 처음에는 하나씩, 필요해질 때 한 범주씩 늘려가는 것이 정석이다.
