# CLAUDE.md: 프로젝트의 두뇌를 만들다

## 개요

CLAUDE.md는 Claude Code와 함께 일할 때 **세션이 바뀌어도 유지되는** 프로젝트 컨텍스트입니다.
일회성 지시가 아니라, "이 프로젝트에서는 항상 이렇게 일해"라는 영구적 설정을 만들 수 있습니다.

예를 들어, PM이 CLAUDE.md에 "항상 수치에는 출처를 붙여", "마케팅 가정을 사실처럼 쓰지 말아" 같은 규칙을 써두면,
이후 모든 Claude 대화에서 자동으로 그 규칙이 적용됩니다.

Claude Code 의 메모리는 두 갈래입니다 — **사람이 쓰는 CLAUDE.md** 와 **Claude 가 스스로 쓰는 auto memory**(5절). 이 장에서는 둘 다 다룹니다.

---

## 1. CLAUDE.md 메모리 계층 구조

Claude Code는 4개 스코프로 CLAUDE.md를 찾고 모두 컨텍스트에 합쳐 적용합니다.
공식 스코프는 **Managed policy(조직) > User(`~/.claude`) > Project(`./` 또는 `./.claude`) > Local(`CLAUDE.local.md`)** 입니다:

```
┌─────────────────────────────────────────────────────┐
│ 1. Managed policy (조직 IT/DevOps가 배포)           │
│    └─ macOS: /Library/Application Support/          │
│       ClaudeCode/CLAUDE.md                          │
│       Linux/WSL: /etc/claude-code/CLAUDE.md         │
│       Windows: C:\Program Files\ClaudeCode\CLAUDE.md │
│    └─ 조직 전체 강제 규칙 (개인 설정으로 제외 불가) │
│       • 회사 코딩 표준, 보안 정책, 컴플라이언스     │
│                                                      │
│ 2. User (~/.claude/CLAUDE.md)                       │
│    └─ 내 모든 프로젝트 공통 개인 선호               │
│       • 문서 작성 스타일                             │
│       • 개인 도구 단축키                             │
│       • 단위 표준 (USD 사용, UTC 시간대)            │
│                                                      │
│ 3. Project (./CLAUDE.md 또는 ./.claude/CLAUDE.md)   │
│    └─ 팀 공유 프로젝트 규칙 (소스 컨트롤로 공유)    │
│       • 제품명, 타겟층, 지표                        │
│       • Persona 정의                                │
│       • 기술 스택 정보                              │
│                                                      │
│ 4. Local (./CLAUDE.local.md) ⭐ 개인만 적용         │
│    └─ 프로젝트별 개인 선호 (.gitignore 추가)        │
│       • 내 샌드박스 URL, 선호 테스트 데이터         │
│       • 깊이 선호도 (think vs ultrathink)           │
└─────────────────────────────────────────────────────┘

로드 순서: 넓은 스코프(Managed policy) → 좁은 스코프(Local) 순으로
컨텍스트에 합쳐지며, 나중에 읽히는 더 구체적인 규칙이 우선합니다.
즉 적용 우선순위는 Local > Project > User > Managed policy.
(단, Managed policy 는 개인 설정으로 제외할 수 없는 강제 규칙)
```

**예시:**

```
User CLAUDE.md (~/.claude): "모든 내용을 영어로 작성"
Project CLAUDE.md (./): "TaskFlow 프로젝트는 한국어로 작성"
→ TaskFlow 프로젝트에서는 한국어 사용 (Project가 User보다 우선)

Project CLAUDE.md: "모든 문서는 정중한 톤"
Local CLAUDE.local.md: "내 작업 중에는 캐주얼 톤으로"
→ 내 로컬 작업에서는 캐주얼 톤 (Local이 Project보다 우선)
```

> **하위 폴더별 규칙은 `.claude/rules/` 로 진화했습니다.** 예전에는 폴더마다 CLAUDE.md를 두는 "Directory 계층" 방식을 썼지만, 지금은 프로젝트 `.claude/rules/` 폴더에 토픽별 `.md` 파일을 두고, frontmatter 의 `paths` 필드로 **특정 파일 패턴에서만 조건부 로드**하는 방식이 표준입니다. (자세한 내용은 아래 1.1 절)

### 1.1 path-scoped 규칙: `.claude/rules/`

큰 프로젝트에서는 규칙을 CLAUDE.md 한 파일에 모으는 대신 `.claude/rules/` 폴더에 토픽별로 나눠둘 수 있습니다.

```text
your-project/
├── .claude/
│   ├── CLAUDE.md           # 메인 프로젝트 규칙
│   └── rules/
│       ├── code-style.md   # 코드 스타일
│       ├── testing.md      # 테스트 규약
│       └── api-design.md   # API 설계 규칙
```

`paths` frontmatter 가 없는 규칙은 매 세션 로드되고, 있으면 **매칭되는 파일을 다룰 때만** 로드됩니다.

```markdown
---
paths:
  - "src/api/**/*.ts"
---

# API 개발 규칙

- 모든 API 엔드포인트는 입력 검증 포함
- 표준 에러 응답 형식 사용
```

이렇게 하면 PM 입장에서 "문서를 다룰 때만 문서 규칙", "마케팅 폴더에서만 마케팅 톤" 처럼 컨텍스트를 필요한 순간에만 켤 수 있어, CLAUDE.md 가 비대해지는 것을 막습니다.

## 2. CLAUDE.md 작성 방법

### 처음부터 완벽하게 만들지 않는다

**원칙**: "같은 내용을 두 번 이상 설명하게 되면 그때 CLAUDE.md로 분리한다"

처음부터 완벽한 CLAUDE.md를 만들려다 시간을 낭비하는 패턴이 흔합니다. 대신 아래 단계를 따릅니다:

1. **세션에서 자연스럽게 대화 시작** — CLAUDE.md 없이 그냥 작업을 시작합니다.
2. **같은 배경 설명을 두 번째 반복하는 시점 → 메모** — "이미 말했는데 또 설명했다"는 순간이 추가 신호입니다.
3. **세션 종료 후 초안 요청** — "오늘 우리가 배운 것 중 문서화해야 할 게 뭐가 있어?"로 Claude에게 초안을 받습니다.
4. **검토 후 CLAUDE.md에 추가** — Claude가 제안한 내용 중 필요한 것만 골라 파일에 기입합니다.

**효과**: 실제로 반복된 맥락만 CLAUDE.md에 쌓이므로 불필요한 내용이 끼어들지 않습니다.

## 3. 실제 CLAUDE.md 작성 예시 (PM 프로젝트용)

### 2.1 Project 레벨 CLAUDE.md

**파일 위치:** `/your-project-root/CLAUDE.md`

**예시 내용:**

```markdown
# TaskFlow - CLAUDE.md
# 제품: 팀 프로젝트 관리 SaaS

## 📌 제품 정의
- **제품명**: TaskFlow
- **설명**: B2B 프로젝트 관리 도구 (Asana/Monday.com 경쟁)
- **타겟 시장**: Series A-C 스타트업, 팀 규모 50-500명
- **홈페이지**: taskflow.io
- **현재 버전**: v2.3.1

## 🎯 북극성 지표 (True North Metric)
- **메인 지표**: Weekly Active Teams (WAT)
  - 현재: 850개 팀
  - 월간 성장률: +12%
  - 목표 (2026 연말): 1,200개 팀
  - 정의: 지난 7일 내 1명 이상이 활동한 팀

- **보조 지표**:
  - Monthly Recurring Revenue (MRR): $47K → $75K (목표)
  - Net Revenue Retention: 105% (업계 기준 100%)

## 👥 Personas

### Persona 1: Sarah - Enterprise Admin
- **역할**: HR/IT 담당자, 회사 인프라 책임
- **나이**: 38세
- **회사**: 500명 스타트업 (Series B)
- **Pain Points**:
  - SSO, SAML 2.0 필수 (보안)
  - 감사 로그 (compliance)
  - 사용자 권한 관리 (RBAC)
- **Success Metrics**:
  - 회사 전체 팀 100개 모두 활동 (WAT +100)
  - 월 결제 $3,500 (팀당 $35)
- **선호 커뮤니케이션**: 이메일, 정기 체크인 콜
- **결정 속도**: 느림 (2-3주 구매 결정)

### Persona 2: Mike - IC Engineer
- **역할**: 백엔드 엔지니어
- **나이**: 28세
- **회사**: 70명 스타트업 (Series A)
- **Pain Points**:
  - GitHub 연동 (작업 자동 연동)
  - 키보드 단축키 (빠른 작업)
  - 빠른 UI (Figma 수준의 응답성)
- **Success Metrics**:
  - TaskFlow에서 1일 평균 12개 작업 처리
  - GitHub pull request와 자동 매칭
- **선호 커뮤니케이션**: Slack, 커뮤니티 포럼
- **결정 속도**: 매우 빠름 (직접 사용 후 자동 채택)

- **역할**: PM/팀 리드
- **나이**: 34세
- **회사**: 180명 스타트업 (Series A 후반)
- **Pain Points**:
  - 팀 가시성 (모든 프로젝트 한눈에)
  - 워크로드 밸런싱 (팀원 과부하 감지)
  - 리포팅 자동화 (월간 보고서)
- **Success Metrics**:
  - TaskFlow로 주간 스탠드업 시간 50% 단축
  - 팀 벌나우트 지표 감지 (과부하 팀원 신속 파악)
- **선호 커뮤니케이션**: 화상 회의, 문서 협력
- **결정 속도**: 중간 (팀과 협의 후 1주일 내)

## 📝 Writing Style (작문 스타일)
- **언어**: 한국어 기본, 기술 용어는 영문 병기
  - 예: "대시보드(Dashboard)를 커스터마이징할 수 있습니다"

- **문서 구조**: 항상 "문제→근거→제안→검증" 순서
```

  ❌ "대시보드 기능을 추가해야 합니다"
  ✅ "팀 리드 3명이 월간 리포트 작성에 4시간씩 소요 중입니다(문제).
     경쟁사(Asana) 대비 리포팅 자동화가 없는 상황입니다(근거).
     대시보드 자동화 기능을 Q2에 추가하면 월 4시간/팀원 절감
     가능합니다(제안). 3개 고객사 파일럿에서 확인됨(검증)."

```
- **수치 사용**: ALWAYS 출처 명시
```

  ❌ "대부분의 팀이 원하는 기능입니다"
  ✅ "고객 인터뷰 5/7명(71%)이 요청한 기능입니다"
  ✅ "2026년 Gartner 보고서에서 RBAC가 구매 결정 상위 3개
     기능으로 선정됨"

```
- **가정 vs 사실**: NEVER 가정을 사실처럼
```

  ❌ "사용자들은 이 기능을 좋아할 것입니다"
  ✅ "가정: 사용자들이 이 기능을 좋아할 것으로 예상됩니다.
     검증 방법: 고객 인터뷰 또는 A/B 테스트"

```
- **Tone & Voice**:
  - 대상이 Sarah(Admin)일 때: 정중하고 신뢰감 있는 톤
  - 대상이 Mike(Engineer)일 때: 직설적, 기술 중심
  - 대상이 Alex(PM)일 때: 데이터 기반, 비즈니스 임팩트 강조

## 📋 기본 규칙

### DO (필수)
- 모든 제안에 근거(데이터/인터뷰/경쟁사분석) 포함
- 수치는 항상 출처 명시
- 기능 영향도는 "구현 복잡도" + "사용자 영향" 2가지 평가
- 새 기능 제안 시 우선순위 근거 제시 (RICE: Reach/Impact/Confidence/Effort)

- 고객 이름, 연락처 외 개인정보 포함 금지

- "나중에", "아마도" 같은 모호한 표현 금지
- 내부 임금, 펀딩 정보, M&A 희설 포함 금지

## 🔄 검토 주기
- **분기별 리뷰**: 이 파일 업데이트 (Q1/Q2/Q3/Q4)
- **팀 확대 시**: Persona 추가/수정
- **제품 피봇**: 새로운 북극성 지표 정의

## 🏷️ 태그 (참고: Claude가 검색할 때 사용)
#product-context #personas #taskflow #saas-pm
```

### 2.2 Directory 레벨 CLAUDE.md (선택사항)

**파일 위치:** `/project/docs/CLAUDE.md`

```markdown
# TaskFlow - 문서 작성 가이드

## 이 폴더의 역할
/docs/ 폴더의 모든 문서는 **대내외 공식 문서**입니다.
따라서 Global + Project CLAUDE.md 규칙을 따르되,
문서 특화 규칙을 추가합니다.

## 문서 타입별 구조

### PRD (Product Requirements Document)
1. 개요 (한 문단, 30초 읽기)
2. 문제 정의 (데이터/인터뷰 근거)
3. 목표 & 성공 기준
4. 사용자 스토리 (Persona별)
5. 기술 요구사항 (엔지니어 리뷰용)
6. 위험 & 의존성
7. 로드맵 & 마일스톤

### 회의록 (Meeting Notes)
- 참석자 (이름 + 직책)
- 결정사항 (Action item 형식)
- 데이터/근거 (논의된 수치)
- 다음 단계 (Owner + 마감일)
- 반박/이견 명시 (의견 불일치 기록)

## 금지 사항 (문서)
- ❌ 개인 의견 (근거 없는)
- ❌ 미래 추측 ("다음달에는...")
- ❌ 경쟁사 루머
- ❌ 임직원 성과 평가 언급

## 외부 공개 가능 여부
- PRD: 요약본만 (마케팅, 고객 공개 불가)
- 회의록: 공개 불가 (결정사항만 별도 공지)
- 리포트: 공개 가능 (고객 사례 제외)
```

## 4. `#` 단축키: 메모리에 빠르게 적립

CLAUDE.md 파일을 직접 열지 않고도, 입력 시작에 `#` prefix 를 붙이면 그 내용이 메모리에 적립됩니다.
"remember …" 처럼 말로 부탁해도 동일합니다.

> **자주 하는 오해 (정정)**: `#` 로 적은 규칙은 "세션 종료 시 사라지는 1회성"이 **아닙니다**. 실제로는 Claude 가 그 내용을 **auto memory 에 영구 저장**합니다. 세션을 새로 열어도 그대로 남아 있습니다(저장 위치·동작은 아래 5절 참고). 진짜로 이번 대화에서만 쓰고 싶다면 `#` 없이 그냥 대화 중에 지시하세요 — 그건 저장되지 않습니다.

**사용 문법:**

```
# [기억시킬 규칙]
> [질문]
```

**예시:**

```
Terminal 입력:

# 앞으로 판매 분석은 항상 "예상치" vs "실제" 2가지로 구분해줘

> @q1-sales-report.csv 이번 분기 판매 트렌드를 분석해줘

Claude Code 응답:

💾 Writing memory…  (auto memory 에 저장됨)

📊 Q1 판매 분석 (예상 vs 실제)

| 월 | 예상 | 실제 | 차이 |
|----|------|------|------|
| 1월 | $250K | $245K | -2% |
| 2월 | $260K | $267K | +3% |
| 3월 | $270K | $289K | +7% |

---

다음 세션에서도 (저장됐으므로 그대로 적용):

> @q1-sales-report.csv 이번 분기 판매 트렌드를 분석해줘

Claude Code 응답:

📊 Q1 판매 분석 (예상 vs 실제)
- 1월: 예상 $250K / 실제 $245K ...

(# 으로 적은 규칙이 메모리에 남아 다음 세션에도 적용됨)
```

**`#` 단축키 vs CLAUDE.md 직접 작성:**

- `#` / "remember …": Claude 가 auto memory 에 저장. 빠르고, 사람이 파일을 안 열어도 됨.
- "이걸 CLAUDE.md 에 추가해줘": 팀 공유용 규칙으로 CLAUDE.md 에 적립(소스 컨트롤로 공유).
- 진짜 1회성: `#` 없이 그냥 지시 → 저장 안 됨, 그 대화에서만 적용.

## 6. Auto memory: Claude가 스스로 적는 메모리

Claude Code 에는 메모리 시스템이 두 가지 있습니다.
하나는 **사람이 쓰는 CLAUDE.md**, 다른 하나는 **Claude 가 스스로 쓰는 auto memory** 입니다.
둘 다 매 세션 시작 시 로드됩니다.

| 구분 | CLAUDE.md | Auto memory |
| --- | --- | --- |
| **누가 쓰나** | 사람(PM) | Claude |
| **무엇이 담기나** | 지시·규칙 | Claude 가 발견한 학습·패턴·선호 |
| **스코프** | 프로젝트/사용자/조직 | 저장소(repo)별, worktree 간 공유 |
| **쓰임새** | 코딩 표준, 워크플로, 아키텍처 | 빌드 명령, 디버깅 인사이트, Claude 가 알아챈 선호 |

Auto memory 는 기본값으로 켜져 있습니다. PM 이 매번 적어두지 않아도, Claude 가 작업 중에 "이건 다음에도 쓸모 있겠다" 싶은 것(반복되는 교정, 선호, 빌드 명령 등)을 스스로 골라 저장합니다. 화면에 "Writing memory" / "Recalled memory" 가 보이면 그 순간 Claude 가 메모리를 쓰거나 읽는 중입니다.

> **요건**: auto memory 는 Claude Code v2.1.59 이상에서 동작합니다.

### 5.1 저장 위치

각 프로젝트마다 별도의 메모리 폴더가 생깁니다:

```text
~/.claude/projects/<project>/memory/
├── MEMORY.md          # 간결한 인덱스, 매 세션 로드됨
├── debugging.md       # 디버깅 패턴 상세 노트
├── api-conventions.md # API 설계 결정
└── ...                # Claude 가 만든 기타 토픽 파일
```

- `<project>` 는 git 저장소 기준으로 결정되며, 같은 repo 의 모든 worktree·하위 폴더가 하나의 메모리 폴더를 공유합니다.
- 매 세션 시작 시 로드되는 것은 `MEMORY.md` 의 **앞 200줄(또는 25KB)** 까지입니다. 그 이상 분량은 시작 시 로드되지 않습니다.
- `MEMORY.md` 는 메모리 폴더의 **인덱스** 역할이고, `debugging.md` 같은 토픽 파일은 필요할 때 Claude 가 직접 읽어옵니다.
- auto memory 는 머신 로컬 — 다른 컴퓨터·클라우드 환경과 공유되지 않습니다.

### 5.2 `/memory` 로 열람·편집

`/memory` 명령은 현재 세션에 로드된 CLAUDE.md·CLAUDE.local.md·rules 파일을 모두 보여주고, auto memory 를 켜고/끄며, auto memory 폴더를 열 수 있게 합니다. 저장된 메모리는 전부 일반 markdown 이라 직접 읽고, 고치고, 지울 수 있습니다.

```text
> /memory

[ 로드된 파일 목록 + auto memory 토글 + 폴더 열기 ]
```

### 5.3 PM 운영 습관: 메모리 주기 감사

Auto memory 는 편하지만, Claude 가 "내가 잘못 이해한 선호"나 더 이상 유효하지 않은 패턴을 저장해둘 수도 있습니다. 그래서 PM 에게 권하는 습관은:

- **주기적으로 `/memory` 로 auto memory 폴더를 열어** Claude 가 무엇을 저장했는지 직접 확인합니다.
- 틀렸거나 낡은 항목은 그 자리에서 지우거나 고칩니다(일반 markdown 이므로 자유롭게 편집).
- "팀 전체가 따라야 할 규칙"이라면 auto memory 가 아니라 CLAUDE.md 로 옮깁니다(소스 컨트롤로 공유되니까).

요약하면 **CLAUDE.md = 내가 쓰는 규칙 / auto memory = Claude 가 쓰는 학습**이고, PM 의 역할은 후자를 가끔 감사해서 신뢰할 만한 상태로 유지하는 것입니다.

## 8. CLAUDE.md 작성 체크리스트

### 첫 작성 시 (필수 섹션)

- [ ] 제품 정의 (이름, 설명, 타겟)
- [ ] 북극성 지표 (1개 메인 + 2-3개 보조)
- [ ] 3-5개 Persona (이름, 역할, Pain Point, Success Metric)
- [ ] Writing Style (톤, 구조, 수치 규칙)
- [ ] DO/NEVER 규칙 (5개씩)
- [ ] 기술 스택 (백엔드, 프론트엔드, 데이터베이스)

### 유지보수 (정기 리뷰)

- [ ] 분기별 1회 (북극성 지표 업데이트?)
- [ ] Persona 변화 감지 (새로운 타겟층?)
- [ ] 기술 스택 변경 기록
- [ ] 금지 항목 추가 (회의록 가정 금지 등)

### 팀 공유

- [ ] 신입 온보딩 시 CLAUDE.md 먼저 읽기
- [ ] 분기별 전체 팀 리뷰 (1시간)
- [ ] Persona 변경 시 모두에게 공지

### 주기적 다이어트: 더 짧을수록 더 잘 읽힌다

**문제**: CLAUDE.md가 길어질수록 Claude가 중요한 규칙을 놓칩니다. 300줄 파일은 매 요청마다 약 4,000 토큰을 소비합니다.

**목표**: 80줄 이하(매 요청 약 800 토큰)로 유지합니다.

**다이어트 방법:**

1. `/revise-claude-md` 슬래시 커맨드로 월 1회 정리 요청
2. "코드만 봐도 알 수 있는 내용"은 삭제 — CLAUDE.md는 non-obvious 규칙만 담습니다
3. 완료된 프로젝트의 오래된 규칙 제거
4. 상세 내용은 `@docs/` 참조로 분리 (Lazy Loading) — CLAUDE.md에는 링크만 남기고, 실제 내용은 별도 파일에 보관합니다

**결과**: 300줄 → 80줄 달성 시 토큰 80% 절감, Claude의 규칙 준수율 향상

### AGENTS.md: 다른 AI 도구에서도 읽힌다

CLAUDE.md와 동일한 내용을 `AGENTS.md` 이름으로도 저장해 두면 멀티툴 환경을 무료로 지원할 수 있습니다.

- OpenAI Codex, GitHub Copilot Workspace 등 다른 에이전트 도구도 `AGENTS.md`를 자동으로 읽습니다.
- 두 파일 중 하나를 정본으로 관리하고 나머지를 복사본으로 유지하면, 추가 관리 부담 없이 어떤 AI 도구에서도 일관된 프로젝트 컨텍스트를 유지할 수 있습니다.

## 9. 실습 과제

1. **자신의 프로젝트 CLAUDE.md 작성 (30분)**

- 제품명, 북극성 지표 정의
- Persona 3개 최소 작성
- Writing Style 규칙 3개 최소

2. **사전 vs 사후 비교 (10분)**

- CLAUDE.md 없이 1개 질문 → Claude에게 요청
- 같은 질문을 CLAUDE.md 작성 후 반복
- 응답 품질 차이 관찰

3. **`#` 메모리 적립 테스트 (5분)**

- 한 가지 특수 분석 스타일을 `#` 으로 적립
- `/memory` 로 auto memory 폴더를 열어 저장됐는지 확인
- 새 세션을 열어 같은 규칙이 그대로 적용되는지(영구 저장) 확인
