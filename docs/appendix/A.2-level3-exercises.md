# 레벨 3 실습: 나만의 Claude Code 워크플로우 설계하기

## 이 실습에 대하여

레벨 1(따라하기)과 레벨 2(변형하기)를 거쳤다면, 이제 자신의 실제 프로젝트에 Claude Code를 적용할 차례입니다.

레벨 3의 핵심은 **"자동화 / 증강 / 직접판단" 분류**입니다. 모든 PM 작업을 이 세 가지로 나누고, Claude Code가 어디에 어떻게 들어가는지를 설계합니다.

---

## 실습 1: 내 업무의 자동화/증강/직접판단 지도 그리기

### 과제

지난 1주일의 업무를 되돌아보고, 아래 표를 채우세요.

| 업무 | 소요 시간 | 분류 | Claude Code 활용 방안 | 예상 시간 절감 |
| --- | --- | --- | --- | --- |
| (예) 주간 리포트 작성 | 2시간 | 자동화 | @data.csv 주간 리포트 생성 | 1.5시간 |
| --- | --- | --- | --- | --- |

### 검증 방법

완성된 표를 Claude Code에 입력하세요:

```bash
> @my-workflow-map.md 이 업무 분류표를 검토해줘.
  1. "증강"으로 분류한 것 중 실제로는 "자동화"가 가능한 것이 있나?
  2. "직접판단"으로 분류한 것 중 Claude가 옵션을 생성해줄 수 있는 것은?
  3. 가장 시간 절감 효과가 큰 TOP 3를 근거와 함께 추천해줘
```

### 성공 기준

- 최소 10개 업무를 분류했는가
- 3가지 카테고리에 각각 최소 2개 이상 있는가
- Claude에게 검증을 받았는가

---

## 실습 2: Discovery 워크플로우 설계

### 과제

현재 담당 중인 제품/기능의 실제 사용자 데이터를 활용하여:

1. **분석할 데이터 소스를 정의하세요**
  - 어떤 파일? (CSV, JSON, 인터뷰 노트?)
  - 어떤 형식?
  - 데이터 품질은?

2. **Claude Code에 줄 프롬프트 3개를 작성하세요**
  - 세그멘테이션 프롬프트
  - 문제 발견 프롬프트
  - 기회 영역 프롬프트

3. **Claude의 분석 결과에서 "PM만이 판단할 수 있는 것"을 미리 예측하세요**
  - Claude는 어디까지 분석할 수 있을까?
  - 어디부터는 도메인 지식이 필요할까?

### 제출물

- 프롬프트 3개 (실행 전)
- 실행 결과 스크린샷
- "Claude가 맞은 것 / 놓친 것 / PM이 추가한 판단" 정리

### 평가 기준

- 프롬프트가 구체적인가? (추상적이지 않은가)
- Claude의 분석 결과가 실행 가능한 인사이트를 제공하는가?
- PM이 추가 판단을 통해 새로운 가치를 더했는가?

## 실습 3: Definition 워크플로우 설계

### 과제

실습 2에서 발견한 문제를 기반으로:

1. **CLAUDE.md에 프로젝트 컨텍스트를 설정하세요**
```markdown
   # [프로젝트명] Claude Code 컨텍스트

   ## 프로젝트 개요
   - 목표:
   - 타겟 사용자:
   - 성공 지표:

   ## Claude의 역할
   - 자동화할 작업: (예: 요구사항 분석, PRD 검토)
   - 증강할 작업: (예: 대안안 생성, 리스크 분석)
   - 직접판단이 필요한 부분: (예: 우선순위, 전략적 선택)
```

2. **소크라틱 대화 방식으로 PRD를 작성하세요**
  - Claude의 질문에 답변 (7-10개)
  - PRD v1.0 작성
  - PRD v1.0 → v2.0 반복

3. **커스텀 서브에이전트 최소 2개를 만들어 리뷰를 받으세요**
```bash
   > @prd-v1.md 이 PRD를 이 두 관점에서 검토해줄 수 있어?

     - 🔧 Engineer Agent
     - 💼 Executive Agent
```

### 제출물

- CLAUDE.md 파일 (프로젝트 컨텍스트)
- 서브에이전트 파일 2개 (.claude/agents/ 또는 별도 파일)
- PRD 최종본
- "리뷰에서 받은 피드백 중 수용한 것 / 거절한 것 + 이유" 정리

### 평가 기준

- 컨텍스트가 Claude에게 충분한가?
- PRD의 반복이 1회 이상 있는가?
- 두 가지 이상 서브에이전트의 피드백이 결과에 반영되었는가?

## 실습 4: Delivery 워크플로우 설계

### 과제

실습 3의 PRD를 기반으로:

1. **바이브 코딩으로 프로토타입 또는 내부 도구를 만드세요**
  - Claude와의 대화로 기술 스택 정하기
  - 초기 구현 (2-3시간)
  - 스크린샷 피드백 루프 시작

2. **최소 3회의 피드백 루프를 거치세요**
  - Loop 1: 초기 구현 → 기본 기능 확인
  - Loop 2: 디자인/UX 개선
  - Loop 3: 모바일 반응형 또는 추가 기능

   각 루프마다:
```bash
   [스크린샷]
   > 이 부분을 다음과 같이 수정해줄 수 있어?
     1. ...
     2. ...
     3. ...
```

3. **Vercel 또는 다른 방법으로 팀에 공유하세요**
```bash
   $ vercel deploy
   # 또는 로컬 개발 서버로 팀원에게 공유
```

### 제출물

- 배포된 URL (또는 로컬 접근 방법)
- 피드백 루프 3회 기록 (요청 → 결과 → 판단)
- 스크린샷 비교 (초기 vs 최종)
- "코드를 모르는 PM이 만들었다"는 것의 장점/한계 회고

### 평가 기준

- 프로토타입이 실행 가능한 상태인가?
- 피드백 루프가 3회 이상 명확히 문서화되었는가?
- 팀이 실제로 접근하고 테스트할 수 있는가?

## 실습 5: Growth 워크플로우 설계

### 과제

현재 진행 중이거나 과거의 A/B 테스트 데이터를 활용하여:

1. **Claude Code로 1차 분석을 수행하세요**
```bash
   > @ab_test_data.csv 이 A/B 테스트를 분석해줘.
     1. 통계적 유의성 (p-value, 신뢰구간)
     2. 세그먼트별 분석
     3. 품질 지표 (부작용 확인)
     4. 최종 권고안
```

2. **Claude의 분석에서 "PM만이 알 수 있는 컨텍스트"를 최소 2개 추가하세요**
```bash
   > 이 분석에 추가 컨텍스트가 있어.
     1. 이 테스트 기간에 마케팅 캠페인이 진행됐어
     2. CS 문의가 35% 증가했는데, 이게 우려사항이야

     이 컨텍스트를 고려하면 어떻게 달라져?
```

3. **의사결정 문서를 작성하세요**
```markdown
   # A/B 테스트 의사결정 문서

   ## 결과 요약
   - 전환율: X% → Y% (+Zpp)
   - p-value: 0.00X

   ## 해석
   - Claude의 분석:
   - PM의 컨텍스트:

   ## 의사결정
   - Ship / Iterate / Kill: [선택]
   - 조건:
   - 다음 단계:
```

### 제출물

- 분석 프롬프트와 Claude 응답
- PM이 추가한 컨텍스트 (Claude가 모르는 것)
- 의사결정 문서 1장

### 평가 기준

- 분석이 통계적으로 명확한가?
- PM의 컨텍스트가 의사결정을 바꿨는가?
- 의사결정이 명확하고 실행 가능한가?

## 종합 실습: 6주 워크플로우 설계서

### 과제

실습 1~5를 바탕으로, **자신의 다음 분기 핵심 프로젝트**에 Claude Code를 어떻게 적용할지 6주 계획을 세우세요.

| 주차 | PM 워크플로우 | Claude Code 활용 | 산출물 | 자동화/증강/직접판단 |
| --- | --- | --- | --- | --- |
| 1주 | Discovery | @user-research.csv 세그멘테이션 분석 | 세그먼트 정의 | 자동화 |
| 2주 | Discovery → Definition | 소크라틱 질문 생성 | 컨텍스트 정리 | 증강 |
| ... |  |  |  |  |

### 성공 기준

- 6주 계획이 현실적인가? (실제로 실행 가능한 수준)
- 각 주차에서 "자동화/증강/직접판단"이 명확히 구분되어 있는가?
- "Claude 없이는 불가능했던 것"이 최소 3개 있는가?

### 제출물

- 6주 계획표 (위 양식)
- 각 주차별 주요 산출물 명단
- 예상 시간 절감 계산 (현재 방식 vs Claude Code 활용 방식)

## 자기 평가 체크리스트

아래 항목 중 8개 이상 체크할 수 있다면, 당신은 AI 네이티브 PM의 시작점에 있습니다.

- [ ] CLAUDE.md를 내 프로젝트에 맞게 작성할 수 있다
- [ ] 커스텀 (서브)에이전트를 만들어 멀티 관점 리뷰를 돌릴 수 있다
- [ ] Claude의 분석 결과에서 "맞는 것"과 "의심스러운 것"을 구분할 수 있다
- [ ] 바이브 코딩으로 프로토타입을 당일 완성할 수 있다
- [ ] A/B 테스트 결과를 Claude로 1차 분석하고, PM 판단을 추가할 수 있다
- [ ] "자동화/증강/직접판단"으로 내 업무를 분류할 수 있다
- [ ] Claude의 제안을 거절할 때 그 이유를 명확히 설명할 수 있다
- [ ] Claude가 모르는 컨텍스트를 사전에 제공하는 습관이 생겼다
- [ ] 팀원에게 Claude Code 활용법을 설명할 수 있다
- [ ] 다음 분기에 Claude Code를 적용할 구체적 계획이 있다

## 피드백 및 지원

각 실습을 완료한 후:

1. **Claude Code와의 최종 검토**
```bash
   > @completed-exercise-1.md 이 실습 결과를 평가해줄 수 있어?
     - 강점: 뭐가 잘 됐나?
     - 개선점: 뭐가 부족한가?
     - 다음 단계: 이걸 어떻게 확장할 수 있나?
```

2. **팀원과 공유 회의**
  - 자신이 만든 산출물 공유
  - 피드백 수집
  - 실제 업무에 적용 가능성 검토

3. **반복적 개선**
  - 첫 완성본은 완벽하지 않아도 됨
  - 피드백을 받고 v2, v3로 개선
  - Claude와 함께 반복하며 품질 향상

## 자주 묻는 질문

**Q: "자동화/증강/직접판단"을 어떻게 구분하나요?**

A:
- **자동화**: Claude가 일관되게 정해진 방식으로 처리할 수 있는 작업 (예: 데이터 분석, 문서 정리)
- **증강**: Claude가 옵션/대안을 생성하고 PM이 선택하는 작업 (예: 접근법 비교, 리스크 분석)
- **직접판단**: Claude가 정보를 제공하지만 PM이 최종 판단하는 작업 (예: 우선순위, 기업 전략)

**Q: Claude가 잘못된 분석을 하면 어떻게 하나요?**

A:
- 항상 "이 결과가 맞나?"라고 의심하는 습관 기르기
- 작은 샘플로 먼저 검증 (전체 데이터 전에)
- Claude에게 "이 결과의 한계/가정은?"을 물어보기
- 팀에 공유하기 전에 한 명 이상에게 검증받기

**Q: 실습을 다 못 완료하면?**

A:
- 완벽함을 기준으로 하지 말 것
- 70% 정도 완료되면 피드백을 받고 개선
- "정말 해야 하는 실습"과 "선택 실습"을 구분
- 우선순위: 실습 1, 2, 5 > 실습 3, 4

**Q: 회사에서 Claude Code 도입을 허용하지 않으면?**

A:
- Claude.ai (웹 버전)로 유사한 워크플로우 가능
- 민감한 데이터만 주의하고, 비용-편익 데이터는 활용
- 팀원들과 함께 실습 → 조직 차원의 도입 요청
- 파일럿 프로젝트로 시작해서 성과 증명
