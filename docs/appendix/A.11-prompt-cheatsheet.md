# PM 프롬프트 치트시트

> **이 문서의 용도**: 본문 곳곳에 흩어진 핵심 프롬프트를 한 페이지로 모았습니다. 지금 터미널에 바로 복사해 쓰는 1~2줄짜리 프롬프트 모음입니다. 각 묶음 끝의 링크를 따라가면 같은 프롬프트의 배경·변형·실패 처리까지 본문에서 확인할 수 있습니다.

---

## 사용법 3가지

- **그대로 복사**: 아래 프롬프트의 `대괄호 부분`만 자신의 상황으로 바꿔 붙여넣으세요.
- **파일 참조 붙이기**: `@파일명` 또는 `@폴더명/`을 앞에 붙이면 그 자료를 읽고 답합니다.
- **깊이 조절**: 어려운 판단에는 끝에 `think harder about`을 덧붙여 더 깊이 사고하게 만드세요.

> 여기 예시는 모두 PM 업무 일반에 쓰는 가상 상황입니다. 자신의 실제 회사·고객 데이터로 바꿔서 쓰세요.

## Discovery — 발견

리서치 데이터에서 패턴을 찾고 경쟁 구도를 정리하는 단계입니다.

| 목적 | 복사해 쓰는 프롬프트 |
| --- | --- |
| 설문 데이터 분석 | `@survey-results.csv 이 설문 데이터를 분석해서 핵심 인사이트 5개로 정리해줘.` |
| 인터뷰 묶음 분석 | `@interviews/ 이 폴더의 인터뷰를 분석해서 공통 페인 포인트를 행동 패턴 기준으로 묶어줘.` |
| 사용자 세그먼트 | `이 데이터로 사용자 세그먼트를 3~4개로 묶고, 각 세그먼트의 특징을 한 줄로 설명해줘.` |
| 경쟁사 비교 매트릭스 | `@competitor-data/ 경쟁사별 가격·기능·타깃 고객을 비교 매트릭스로 만들고 강점·약점을 정리해줘.` |
| 멀티소스 병렬 분석 | `에이전트 3개로 경쟁사 자료를 동시에 분석한 뒤, 마지막에 결과를 하나로 종합해줘.` |

## Definition — 정의

문제를 정의하고 PRD·전략 문서로 굳히는 단계입니다.

| 목적 | 복사해 쓰는 프롬프트 |
| --- | --- |
| PRD 질문부터 시작 | `[기능명] PRD를 작성하려고 해. 먼저 내가 놓치고 있을 질문 5~7개를 해줘.` |
| 맥락 붙여 PRD 작성 | `@company-context.md @user-research/ 이 자료로 PRD 초안을 작성해줘.` |
| 핵심 문제 진단 | `이 자료들을 종합해서 우리가 직면한 핵심 문제 하나를 진단해줘.` |
| 전략 방향 제안 | `위 진단에 기반해서 서로 다른 전략 방향 3가지를 제안하고 각각 설명해줘.` |
| 전략 사전 검증 | `이 전략을 think harder about 해줘. 실패할 수 있는 시나리오 5개를 만들어줘.` |

## 막힐 때 쓰는 메타 프롬프트

프롬프트가 잘 안 먹힐 때, 다음 한 줄이 종종 더 효과적입니다.

- **질문을 시키기**: `바로 답하지 말고, 먼저 나에게 필요한 질문부터 해줘.`
- **가정 드러내기**: `네가 지금 깔고 있는 가정 3가지를 먼저 말해줘.`
- **반박 요청**: `이 결론의 가장 약한 부분을 비판해줘.`
- **형식 고정**: `결과를 표 한 개로만 정리해줘.`
- **범위 좁히기**: `한 번에 다 하지 말고, 첫 단계만 먼저 해줘.`

## CoT 프롬프팅: Claude가 단계별로 생각하게 만들기

CoT(Chain-of-Thought)는 Claude에게 결론을 바로 내리지 말고 단계별로 추론하도록 요청하는 기법입니다.
복잡한 분석이나 설계 결정에서 결과 품질이 크게 높아집니다.

### 기본 패턴

| ❌ 단순 요청 | ✅ CoT 요청 |
|---|---|
| "이 아키텍처가 맞나?" | "이 아키텍처의 장단점을 단계별로 분석해줘" |
| "어떤 DB가 좋아?" | "요구사항을 고려해서 DB 선택지를 심사숙고해줘" |
| "에러 고쳐줘" | "에러 원인을 단계별로 추론한 후 해결 방안을 제시해줘" |

### CoT 트리거 표현

- "심사숙고해줘"
- "단계별로 생각해줘"
- "추론 과정을 보여줘"
- "결론 전에 논거를 먼저 나열해줘"

> **참고**: `ultrathink` 키워드는 모델이 더 깊게 추론하도록 하는 공식 Claude 키워드입니다.
> `think`, `think hard`, `think harder`는 일반 텍스트로 전달되므로 효과가 없습니다.

## 사고와 답변 분리: `<thinking>`/`<answer>` 태그 패턴

Claude의 추론 과정과 최종 답변을 구조적으로 분리하고 싶을 때 태그를 명시적으로 요청합니다.

### 요청 방법

```
"REST API를 구축하려면 어떻게 해야 해?
생각 과정을 <thinking> 태그에 입력해주고,
그것을 분석한 결과를 <answer> 태그에 답변해줘."
```

### 출력 예시

```xml
<thinking>
REST API 구축 시 고려할 요소:
1. 프레임워크 선택: Express vs Fastify vs Hono (경량화, 타입스크립트 지원)
2. 인증 방식: JWT vs Session (무상태 vs 서버 세션)
3. DB 연결: ORM(Prisma) vs 직접 쿼리
4. 에러 처리: 표준 에러 코드 정의 필요
...
</thinking>

<answer>
PM 관점에서 권장하는 구조:
- 프레임워크: Express.js (생태계 최대, 레퍼런스 풍부)
- 인증: JWT (서버리스 환경 호환)
...
</answer>
```

### 언제 쓰나

- **복잡한 기술 결정**: 아키텍처, DB 선택, 라이브러리 비교
- **긴 분석 작업**: 경쟁사 분석, 유저 리서치 합성
- **검증이 중요한 작업**: 추론 과정이 맞는지 확인하고 싶을 때
- **교육 목적**: Claude가 왜 그 결론에 도달했는지 이해하고 싶을 때

## SPEC.md: 기술 명세를 함께 만들기

PRD.md가 "무엇을 만들고 왜"를 담는다면, SPEC.md는 "어떻게 만드는가"를 담습니다.

### SPEC.md 필수 구성 요소

```markdown
# SPEC.md

## 기술 스택
- Frontend: Next.js 14 (App Router)
- Backend: FastAPI / Express.js
- Database: Supabase (PostgreSQL)
- Infra: Vercel + Cloudflare Workers
- ML: (해당 시) Anthropic API

## 주요 변경 내역 (ADR)
<!-- Architecture Decision Record -->
### ADR-001: DB 선택
- 결정: Supabase
- 이유: RLS 내장, 실시간 구독, Auth 포함
- 대안: PlanetScale, Neon

## 아키텍처 (ARCHITECTURE)
[Mermaid 다이어그램 또는 ASCII로 시스템 구조]

## 디자인 (DESIGN)
[ASCII 목업 또는 Figma 링크]
```

### Claude Code에서 SPEC.md 만들기

```
"현재 PRD.md를 기반으로 SPEC.md를 작성해줘.
기술 스택, ADR, 아키텍처 다이어그램(Mermaid)을 포함해줘.
아직 코드는 작성하지 말아줘"
```

PRD.md → SPEC.md 순서로 작성하면, 개발 시작 전에 기술 방향을 확정하고 Claude Code에게 일관된 컨텍스트를 줄 수 있습니다.
