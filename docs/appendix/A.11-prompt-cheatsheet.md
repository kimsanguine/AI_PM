# PM 프롬프트 치트시트

> **이 문서의 용도**: 본문 곳곳에 흩어진 핵심 프롬프트를 한 페이지로 모았습니다. 지금 터미널에 바로 복사해 쓰는 1~2줄짜리 프롬프트 모음입니다. 각 묶음 끝의 링크를 따라가면 같은 프롬프트의 배경·변형·실패 처리까지 본문에서 확인할 수 있습니다.

---

## 사용법 3가지

- **그대로 복사**: 아래 프롬프트의 `대괄호 부분`만 자신의 상황으로 바꿔 붙여넣으세요.
- **파일 참조 붙이기**: `@파일명` 또는 `@폴더명/`을 앞에 붙이면 그 자료를 읽고 답합니다.
- **깊이 조절**: 어려운 판단에는 끝에 `think harder about`을 덧붙여 더 깊이 사고하게 만드세요.

> 여기 예시는 모두 PM 업무 일반에 쓰는 가상 상황입니다. 자신의 실제 회사·고객 데이터로 바꿔서 쓰세요.

## Discovery — 발견

리서치 데이터에서 패턴을 찾고 경쟁 구도를 정리하는 단계입니다.

| 목적 | 복사해 쓰는 프롬프트 |
| --- | --- |
| 설문 데이터 분석 | `@survey-results.csv 이 설문 데이터를 분석해서 핵심 인사이트 5개로 정리해줘.` |
| 인터뷰 묶음 분석 | `@interviews/ 이 폴더의 인터뷰를 분석해서 공통 페인 포인트를 행동 패턴 기준으로 묶어줘.` |
| 사용자 세그먼트 | `이 데이터로 사용자 세그먼트를 3~4개로 묶고, 각 세그먼트의 특징을 한 줄로 설명해줘.` |
| 경쟁사 비교 매트릭스 | `@competitor-data/ 경쟁사별 가격·기능·타깃 고객을 비교 매트릭스로 만들고 강점·약점을 정리해줘.` |
| 멀티소스 병렬 분석 | `에이전트 3개로 경쟁사 자료를 동시에 분석한 뒤, 마지막에 결과를 하나로 종합해줘.` |

## Definition — 정의

문제를 정의하고 PRD·전략 문서로 굳히는 단계입니다.

| 목적 | 복사해 쓰는 프롬프트 |
| --- | --- |
| PRD 질문부터 시작 | `[기능명] PRD를 작성하려고 해. 먼저 내가 놓치고 있을 질문 5~7개를 해줘.` |
| 맥락 붙여 PRD 작성 | `@company-context.md @user-research/ 이 자료로 PRD 초안을 작성해줘.` |
| 핵심 문제 진단 | `이 자료들을 종합해서 우리가 직면한 핵심 문제 하나를 진단해줘.` |
| 전략 방향 제안 | `위 진단에 기반해서 서로 다른 전략 방향 3가지를 제안하고 각각 설명해줘.` |
| 전략 사전 검증 | `이 전략을 think harder about 해줘. 실패할 수 있는 시나리오 5개를 만들어줘.` |

## Delivery — 전달

프로토타입을 만들고 시각 자산을 뽑고 배포하는 단계입니다.

| 목적 | 복사해 쓰는 프롬프트 |
| --- | --- |
| 프로토타입 구성 | `이 구조로 진행해줘. 초기 프로젝트 구성도 자동으로 해줘.` |
| 로컬 미리보기 | `개발 서버를 실행해줘.` |
| 페르소나 이미지 | `우리 제품의 타깃 페르소나 3명에 대한 초상화를 만들어줘.` |
| 사용자 여정 시각화 | `온보딩 플로우의 사용자 여정을 단계별 감정·문제·해결책과 함께 시각화해줘.` |
| 커밋·배포 | `이 변경을 커밋하고, main 브랜치에 병합한 뒤 배포까지 진행해줘.` |

관련 본문: [6.1 바이브 코딩](../part6-delivery/6.1-delivery-vibe-coding.md) · [6.2 시각 자산 생성](../part6-delivery/6.2-delivery-visual-assets.md) · [6.3 GitHub·배포](../part6-delivery/6.3-delivery-github-deploy.md)

## Growth — 성장

실험 결과를 해석하고 지표 체계를 자동화하는 단계입니다.

| 목적 | 복사해 쓰는 프롬프트 |
| --- | --- |
| A/B 테스트 분석 | `@ab_test_results.csv A/B 테스트 결과를 분석하고 통계적으로 유의한지 알려줘.` |
| 세그먼트별 재분석 | `이 A/B 테스트 결과를 세그먼트별로 다시 분석해줘.` |
| 리텐션 관점 전환 | `같은 결과를 전환율이 아닌 7일·14일·30일 리텐션 기준으로 다시 분석해줘.` |
| KPI 체계 설계 | `우리 제품의 KPI 체계를 설계하고, 선행·후행 지표를 구분해줘.` |
| 지표 자동화 | `KPI 계산을 자동화하는 스크립트를 만들어줘.` |

관련 본문: [7.1 실험 분석](../part7-growth/7.1-growth-experiment-analysis.md) · [7.2 KPI 대시보드](../part7-growth/7.2-growth-kpi-dashboard.md)

## 막힐 때 쓰는 메타 프롬프트

프롬프트가 잘 안 먹힐 때, 다음 한 줄이 종종 더 효과적입니다.

- **질문을 시키기**: `바로 답하지 말고, 먼저 나에게 필요한 질문부터 해줘.`
- **가정 드러내기**: `네가 지금 깔고 있는 가정 3가지를 먼저 말해줘.`
- **반박 요청**: `이 결론의 가장 약한 부분을 비판해줘.`
- **형식 고정**: `결과를 표 한 개로만 정리해줘.`
- **범위 좁히기**: `한 번에 다 하지 말고, 첫 단계만 먼저 해줘.`
