# 관통 시나리오: PlanMyDay — 하나의 제품으로 보는 PM 워크플로우 전체

## 이 시나리오에 대하여

Part 2-5에서는 PM의 각 단계별 기술과 Claude Code 활용법을 독립적으로 배웁니다. 하지만 실제 PM의 일은 연결되어 있습니다. Discovery에서 발견한 문제가 Definition의 PRD로 변환되고, 그 PRD가 Delivery의 프로토타입 방향을 결정하며, 마침내 Growth 단계의 실험 설계까지 영향을 미칩니다.

이 시나리오는 **하나의 제품 PlanMyDay가 6주간 겪는 실제 PM 워크플로우**를 보여줍니다. 각 단계에서 Claude Code를 어떻게 활용하고, PM이 어디서 판단을 내리며, 언제 실패할 수 있는지 경험해보세요.

---

## 제품 소개: PlanMyDay

**PlanMyDay**는 직장인의 업무 과부하를 해결하는 AI 기반 일일 계획 SaaS입니다.

- **타겟**: 30-50대 직장인, 일일 평균 20+ 태스크 처리
- **핵심 기능**: 일정/이메일/메신저로부터 자동 태스크 추출 → 우선순위 AI 정렬 → 1일 실행 계획 생성
- **현황**:
  - 출시 6개월 전
  - 12,000 MAU, 3,000 PAU
  - 30일 리텐션: 34% → 28%로 급락 (지난 2개월)
  - NPS: 32 → 24로 하락
  - 팀 규모: PM 1명, 엔지니어 3명, 디자이너 1명

**위험 신호**: 초기 가입은 잘되지만, 첫 2주 안에 사용자의 66%가 떠나가는 상황. 원인을 찾아야 합니다.

## Stage 1: Discovery — "왜 사용자가 떠나고 있는가?"

### 상황

지난 2개월간 30일 리텐션이 34%에서 28%로 하락했습니다. 팀은 불안합니다. 원인이 뭘까요?

- 지난주 실시한 NPS 서베이: 32 → 24 (이탈 신호 강화)
- 이탈 사용자 30명 심층 인터뷰 완료
- 사용자 행동 데이터 (로그 데이터, 피처 사용률): CSV 형태로 정제됨

### Claude Code 워크플로우

PM이 Claude Code를 띄웁니다. 데이터가 준비되어 있습니다.

```
> @nps-survey-q4.csv @user-interviews-jan/
  이 두 데이터를 종합 분석해줘.

  1. NPS 하락의 주요 원인 3가지 (정량 + 정성 교차 분석)
  2. 사용자 세그먼트별 이탈 패턴 (언제, 왜?)
  3. "떠나려는 사용자"와 "남아있는 사용자"의 행동 차이
```

Claude가 분석을 시작합니다. 몇 분 후:

```
📊 분석 결과

[이탈 원인 Top 3]

1. 습관 형성 실패 (42% of churned users)
   - Day 3-7 사이에 이탈 집중
   - 패턴: 가입 후 2-3회 사용 → 반복 일정 미설정 → 방문 중단
   - 인터뷰 증거: "처음엔 좋았는데, 매일 할 일을 다시 입력하기 싫었어"

2. 복잡한 초기 설정 (28% of churned users)
   - 캘린더/이메일 연동 과정에서 이탈
   - 특히 회사 계정(Outlook) 연동 실패율 높음
   - 인터뷰: "설정이 너무 복잡했어. 3분만에 포기했어"

3. 명확한 가치 미인지 (30% of churned users)
   - "왜 써야 하는지 모르겠어요"
   - 직급/연령별로 다른 니즈

[세그먼트 분석]

떠난 사용자:
- 평균 사용 기간: 4.2일
- 평균 매일 사용 일수: 2.1일
- 반복 일정 설정 사용자: 8%

남아있는 사용자:
- 평균 사용 기간: 42일
- 평균 매일 사용 일수: 5.2일
- 반복 일정 설정 사용자: 78%

→ 명확한 패턴: 반복 일정을 설정하는 사용자는 거의 모두 남아있다
```

### PM 판단 포인트

PM이 리드백 노트를 펼칩니다. 세 가지 옵션이 떠올랐습니다:

1. **습관 형성 실패** (42%) → 해결하면 리텐션 +8-10% 예상
2. **복잡한 설정** (28%) → 해결하면 리텐션 +5-7% 예상
3. **명확한 가치 미인지** (30%) → 더 복잡한 오너십 모델 필요

PM은 다음을 고려합니다:
- 신규 사용자 중 42%가 Day 7에 떠나가는 것은 **즉시성** 문제
- 반복 일정 설정이 가장 강한 상관관계 (78% vs 8%)
- ROI: 설정 개선보다 **Day 7 습관 형성 개선**이 더 빠른 효과

**PM 결정**: "**Day 7 습관 형성**을 우선 해결하자. 반복 일정을 설정하도록 유도하는 UX를 만들어야 한다."

### ⚠️ 이럴 때는 이렇게

이건 실패 케이스입니다. Claude가 인터뷰 노트의 감정 표현을 과해석했어요:

```
[인터뷰 원문]
"솔직히 처음엔 좋다고 생각했는데, 매일 할 일을 다시 입력하기 싫었어요.
일단 바빴기도 하고... 그 다음에 깜빡했어요."

[Claude의 초기 해석]
→ "사용자가 제품 자체에 불만족. UI 개선 필요"

[PM의 재검토]
→ 아니다. 사용자는 "가치"를 느꼈어. 문제는 **지속성**이다.
   "매일 입력" 대신 "한 번 설정하고 반복"으로 가면 해결된다.
```

**교훈**: Claude가 감정 데이터를 해석할 때, PM은 **원문을 항상 검증**해야 합니다. 감정과 근본 원인은 다를 수 있습니다.

## Stage 2: Definition — "Day 7 습관 형성을 어떻게 해결할 것인가?"

### 상황

Discovery에서 얻은 명확한 인사이트:
- **문제**: 신규 사용자 42%가 Day 7에 이탈
- **근본 원인**: 반복 일정 미설정 (규칙적 사용 이탈)
- **솔루션 방향**: "반복 일정 설정을 쉽게"

이제 PRD를 작성해야 합니다. 어떻게 할까요?

### Claude Code 워크플로우

PM은 먼저 Socratic 대화를 시작합니다 (Part 3.1 기법):

```
> @CLAUDE.md @discovery-findings.md

  Day 7 습관 형성 문제를 해결하는 PRD를 작성하려고 해.
  먼저 내가 놓치고 있을 수 있는 질문 5개를 해줘.
```

Claude의 응답:

```
🤔 PRD 작성 전에 고려할 핵심 질문

1. "습관 형성"의 정의가 뭐야?
   - Day 7 연속 사용? Day 7까지 N회 이상 사용?
   - 아니면 "반복 일정 설정 → 그것을 실행"까지?

2. Day 7이 왜 critical인가?
   - 행동 심리학에서 일반적인 습관 형성 기간이 21일인데,
     왜 Day 7에 집중할까?
   - 비즈니스적으로는 Day 7까지 사용자를 잡으면 충분한가?

3. 반복 일정 설정 이외에 "습관"을 측정할 다른 지표는?
   - 반복 일정 설정률만으로는 충분한가?
   - 예: 매일 08:00 플래너 실행, 평균 체크아웃 시간 등?

4. 현재 반복 일정 설정 과정의 정확한 문제점은?
   - UX가 복잡한가? 가치가 명확하지 않은가?
   - 아니면 사용자가 "반복 일정"이라는 개념 자체를 모르는가?

5. "Day 5 vs Day 3"에서 제안하려면?
   - 너무 일찍 제안하면 사용자가 아직 제품 가치를 못 느껴
   - 너무 늦으면 이미 떠났다
   - 최적 타이밍은 뭐야?
```

PM이 각각을 답변합니다:

```
> 좋은 질문들이다. 답변할게:
>
> 1. 습관 형성 = "반복 일정 설정 + 첫 주에 최소 3회 이상 실행"
> 2. Day 7이 critical인 이유: 데이터에서 Day 7-8에 이탈 급증.
>    아마 "첫 주 금요일" 시점에서 사용자가 "이게 필요한가"를 판단하는 것 같다.
> 3. 추가 지표: 매주 반복 일정 완료율 (completion rate)
> 4. 현재 문제는 "UX 복잡성" + "가치 미인지" 혼합
> 5. 최적 타이밍: Day 5가 적당. 충분한 데이터가 모여있고, 아직 떠나기 전이다.
>
> 이제 PRD 초안을 작성해줘.
> 핵심 솔루션: "스마트 루틴 제안"
>   - Day 5에 사용자의 지난 5일 일정 패턴을 분석
>   - "당신은 이런 반복 작업을 하시나요?" 제안 카드 표시
>   - 원클릭으로 반복 일정 생성 (기존 5-단계 vs 1-단계)
```

Claude가 PRD 초안을 생성합니다:

```
# PRD: 스마트 루틴 제안 (Smart Routine Suggestions)

## 목표
Day 5 신규 사용자 리텐션 34% → 42% 달성 (Day 7 기준)

## 문제 정의
신규 사용자의 42%가 Day 7 이전에 이탈. 원인: 반복 일정 미설정으로 인한 일일 입력 부담

## 솔루션 개요
**Smart Routine 카드**: Day 5에 사용자의 지난 5일 일정 데이터를 기반으로 3개의 루틴을 자동 제안

### 제안 로직 (MVP)
- 지난 5일간 입력된 태스크 분석
- 패턴 매칭: "매주 월요일 10시 전략 회의", "매일 12시 점심시간" 등
- 상위 3개 추천 (초기 테스트: 3개가 최적 → 나중에 조정)

### UX 플로우
1. Day 5 11:00 (사용자 평균 로그인 시간)
2. 홈 화면에 카드 표시: "당신의 반복 업무를 제안합니다"
3. 3개 루틴 표시, 각각 "추가하기" / "나중에" 버튼
4. "추가하기" 클릭 → 즉시 반복 일정 생성

## 성공 지표
- 카드 노출 후 클릭률: 40% 이상
- 루틴 추가율 (proposed → accepted): 30% 이상
- Day 7 리텐션: 42% 이상

## 실행 계획
- Sprint 1-2: 스마트 루틴 카드 UX + 제안 로직 (규칙 기반)
- Sprint 3: 배포 + 모니터링
```

이제 PM은 멀티에이전트 리뷰를 요청합니다:

```
> /agents/engineer-reviewer 이 PRD의 기술 실현성을 리뷰해줘.
>   - 제안 알고리즘의 기술 난이도
>   - 백엔드 구현 예상 시간
>   - 위험 요소

> /agents/exec-reviewer 이 PRD의 비즈니스 임팩트를 평가해줘.
>   - 예상 리텐션 개선 현실성
>   - 시장 경쟁력 영향
>   - 다른 전략과의 우선순위
```

### PM 판단 포인트

리뷰 결과:

| 관점 | 의견 | 근거 |
| --- | --- | --- |
| **엔지니어** | 추천 알고리즘은 2스프린트 필요 | 데이터 전처리 + 패턴 매칭 로직 + QA |
|  | 위험: Day 5 정확히 게시 어려울 수 있음 | 시간대 최적화 복잡 |
| **임원진** | Day 5는 너무 이르다 → Day 7 추천 | 사용자가 Day 5에 아직 제품 가치를 못 느낄 수 있음 |
|  | 대신 Day 7 제안하면 "마지막 기회" 심리 효과 | 실제 행동 데이터와 충돌 위험 |

**PM 결정**:
```
1. Day 5 유지 (데이터 기반: Day 7-8에 이탈 급증)
2. Day 3에 먼저 온보딩 제안으로 보충
3. MVP는 규칙 기반 (Day 3: "반복 일정 설정 필요성" 설명)
4. Phase 2에서 ML 기반 제안 추가
5. 타이밍 조정: "Day 5 08:00 or 18:00 중 최근 로그인 시간"
```

### ⚠️ 이럴 때는 이렇게

Claude의 초안 PRD가 15페이지였습니다. 상세하지만 엔지니어들이 읽지 않을 정도.

```
> Claude, 이 PRD가 너무 길어. 엔지니어가 10분 안에 읽을 수 있는
  분량으로 줄여줘. 핵심만.
```

Claude가 압축합니다:

```
최종 PRD (2페이지):

## 스마트 루틴 제안

**문제**: 신규 사용자 42% Day 7 이탈 (반복 일정 미설정)

**솔루션**: Day 5에 사용자 패턴 기반 3개 루틴 제안 카드

**구현**:
- 지난 5일 태스크 분석 → 상위 3개 패턴 추출
- 홈 화면에 카드 표시 (원클릭 추가)

**성공 지표**:
- 카드 클릭률 40%+
- 루틴 추가율 30%+
- Day 7 리텐션 +8%p

**일정**: 2스프린트 + 모니터링

**위험**: Day 5 정확한 배포, 알고리즘 정확도
```

**교훈**: PM은 Claude의 상세함을 관리하고, 청중에 맞게 조정할 수 있어야 합니다.

## Stage 3: Delivery — "직접 프로토타입으로 보여주기"

### 상황

PRD가 확정되었습니다. 하지만 회사의 디자이너는 동시에 3개 프로젝트를 담당 중입니다. 2주를 기다릴 여유가 없습니다.

PM 판단: "스마트 루틴 제안 UI의 프로토타입을 직접 만들어서 엔지니어와 초기 사용자에게 보여주자. 확신을 얻으면 디자이너와 협력할 때 더 빠를 거야."

### Claude Code 워크플로우

PM이 Vibe Coding (Part 4 기법)을 시작합니다:

```
> 스마트 루틴 제안 기능의 프로토타입을 만들어줘.
  Next.js + Tailwind CSS로.

  요구사항:
  - Day 5 이후에 사용자 홈 화면에 카드 표시
  - 사용자의 최근 5일 일정 패턴을 기반으로 3개 루틴 제안
    예: "매주 월요일 10:00 전략 회의"
  - 각 루틴에 "추가하기"와 "나중에" 버튼
  - 모바일 우선 디자인 (375px 기준)
  - 옅은 파란색 배경, 명확한 CTA
```

Claude가 프로토타입을 생성합니다. PM은 screenshot을 받습니다.

```

다음 Discovery 사이클로...
```

스마트 루틴 기능은 이 연결을 완벽하게 따릅니다:
- Discovery: "반복 일정 미설정 = 이탈 원인"
- Definition: "Day 5에 반복 일정 제안하자"
- Delivery: "이렇게 생긼다" (시각화)
- Growth: "효과를 측정하고 다음 세그먼트로"

### 2. PM의 일관된 판단 축이 핵심

매 단계에서 Claude의 제안과 PM의 판단이 달랐습니다:

| 단계 | Claude의 제안 | PM의 판단 | 이유 |
| --- | --- | --- | --- |
| Discovery | "복잡한 설정도 28% 이탈 원인" | "습관 형성 42% 우선" | ROI & 속도 |
| Definition | "Day 7이 더 영향 클 것" | "Day 5 유지" | 데이터 기반 |
| Delivery | "10개 루틴 제안 가능" | "3개 유지" | 사용자 테스트 |
| Growth | "36-50도 효과 있을 것" | "25-35 집중 + 36-50 R&D" | 통계 유의성 |

Claude는 빠르고 넓은 관점을 제공합니다. **PM은 깊이 있는 맥락 판단을 합니다.**

### 3. 속도: 전통적 사이클의 3배

전통적 프로세스 (컨설턴트 + 에이전시):
- Discovery: 2-3주
- Definition: 2-3주
- Delivery: 4-6주
- Growth: 2-3주
- **총 10-15주**

Claude Code 사용 프로세스:
- Discovery: 3-4일
- Definition: 2-3일
- Delivery: 3-4일
- Growth: 2-3일
- **총 4-5주**

**차이점**: 자동화된 데이터 분석, 프로토타입 구현, 대시보드 설계.

### 4. Claude.md의 누적 효과

프로젝트 컨텍스트를 한 곳에 유지하면:
- Claude가 반복 학습 (이전 결정의 이유 이해)
- PM이 일관성 유지 (같은 원칙으로 판단)
- 팀 전체가 같은 페이지 (공유 문서 역할)

### 5. 조심할 점들

✓ Claude의 데이터 분석은 정말 빠르다. 하지만 **원문 검증**하세요.
✓ Claude의 PRD는 정말 상세하다. 하지만 **청중에 맞게 압축**하세요.
✓ Claude의 프로토타입은 정말 빠르다. 하지만 **실제 환경에서 테스트**하세요.
✓ Claude의 통계 분석은 정말 깔끔하다. 하지만 **p-value와 표본 크기 확인**하세요.
