# Claude 유즈케이스 × PM 실전 시나리오

> **핵심 요약**: PM의 일상 업무에 맞게 재해석한 실전 시나리오 모음. 각 시나리오는 "어떤 상황에서 → 어떻게 쓰고 → PM이 무엇을 판단하는지"를 워크스루 형식으로 보여줍니다.

---

## 이 문서의 구성

 Discovery → Definition → Delivery → Growth 워크플로우와 **겹치지 않는** 6가지 새로운 PM 시나리오를 다룹니다.

| 시나리오 | 유즈케이스 | PM 업무 | 활용 도구 |
| --- | --- | --- | --- |
| 1. 시장 규모 산정 | Size a market using your research | 새 시장 진입 의사결정 | Cowork + 웹 리서치 |
| 2. 분산 소스 인사이트 덱 | Source insights to build a deck | 경영진 보고용 덱 구성 | MCP + 멀티소스 |
| 3. 크로스채널 피드백 합성 | Surface themes from feedback | VOC 테마 추출 및 우선순위 | Slack/Notion 연동 |
| 4. 파트너/인수 실사 | Evaluate a company | 파트너십/M&A 실사 보고서 | 공개 데이터 + 분석 |
| 5. 멀티소스 데이터 결합 분석 | Build analysis from charts + data | 분산 데이터 통합 분석 | 브라우저 + 로컬 파일 |
| 6. 대시보드 지표 추출 자동화 | Pull metrics from dashboards | 주간 지표 자동 수집 | Claude in Chrome |

## 시나리오 1: 시장 규모 산정 (TAM/SAM/SOM)

### 상황

당신은 AI 더빙 서비스의 PM입니다. CEO가 "동남아 시장 진출을 검토하고 있는데, 시장 규모를 파악해달라"고 요청했습니다. 다음 주 이사회에서 발표해야 합니다.

### 기존 방식 vs Claude 방식

```
기존 (3~5일):
1. 리서치 펌 리포트 구매 검토 ($2,000~$10,000)
2. 공개 데이터 수집 (Statista, 정부 통계, 업계 리포트)
3. 스프레드시트에서 수동 계산
4. 가정(assumptions) 정리
5. PPT 슬라이드 제작

Claude (3~4시간):
1. 리서치 지시 → 데이터 수집 → 모델 생성 → 검증 → 슬라이드 초안
```

### 워크스루

**Step 1: 리서치 프레임워크 설정**

```
Terminal 입력:
> 동남아 AI 더빙 시장의 TAM/SAM/SOM을 산정하려고 해.
  다음 프레임워크를 따라줘:

  1. TAM (Total Addressable Market):
     - 동남아 전체 미디어 로컬라이제이션 시장
     - 소스: 공개 리포트, 정부 통계, 업계 추정

  2. SAM (Serviceable Addressable Market):
     - 그 중 AI 기반 더빙이 대체 가능한 영역
     - 제외: 라이브 통역, 자막 전용 수요

  3. SOM (Serviceable Obtainable Market):
     - 우리가 3년 내 현실적으로 확보 가능한 점유율
     - 고려: 현재 경쟁 구도, 우리 진입 역량

  각 단계에서 사용한 데이터 소스와 가정(assumptions)을
  명확히 구분해서 표시해줘.

Claude Code 응답:
📊 동남아 AI 더빙 시장 규모 산정을 시작합니다.

- 각국 통계청: 미디어 산업 규모
```

**Step 2: 가정 검증 및 시나리오 분석**


> **💡 PM 판단 포인트**: 시장 규모 산정에서 가장 중요한 것은 "숫자의 정확성"이 아니라 "가정의 투명성"입니다. Claude가 명시한 가정들을 하나씩 검증하고, 이사회에서 "이 숫자가 틀릴 수 있는 지점"을 미리 설명할 수 있어야 합니다.

## 시나리오 2: 분산 소스에서 인사이트를 모아 경영진 덱 만들기

### 상황

분기 사업 리뷰(QBR) 발표를 준비 중입니다. 필요한 데이터가 Notion(전략 문서), Linear(개발 진행률), Slack(팀 논의), Google Sheets(지표)에 흩어져 있습니다. 내일까지 CEO에게 보여줄 10장짜리 슬라이드가 필요합니다.

### 워크스루

**Step 1: MCP로 분산 소스 일괄 수집**

```
Terminal 입력:
> QBR 발표 자료를 만들어야 해. 다음 소스에서 데이터를 수집해줘:

  1. Notion "Q4 전략 문서"에서 목표와 Key Results
  2. Linear "Sprint 보드"에서 이번 분기 완료/미완료 항목
  3. Slack #product-updates 채널에서 이번 분기 주요 릴리즈
  4. @./data/q4-metrics.csv 에서 핵심 지표 트렌드

  각 소스별로 핵심 내용을 정리한 뒤,
  "목표 대비 달성률"을 축으로 통합해줘.

Claude Code 응답:
🔗 4개 소스에서 데이터를 수집합니다...

### Notion — Q4 전략 문서
- OKR 1: [사용자 리텐션 70% 달성] → 현재 68% (97% 달성)
- OKR 2: [신규 시장 3개 진출] → 2개 완료, 1개 진행 중
- OKR 3: [NPS 45점 달성] → 현재 42점 (93% 달성)

### Linear — Sprint 보드
- 계획된 에픽 12개 중 9개 완료 (75%)
- 미완료 3개: [에픽A] 기술 부채, [에픽B] 외부 의존성, [에픽C] 스코프 변경

### Slack — #product-updates
- 주요 릴리즈 5건 식별
- 가장 반응이 좋았던 기능: [기능X] (🎉 반응 23개)
- 논란이 있었던 결정: [결정Y] (스레드 47개)

### CSV — 핵심 지표
- MAU: 12만 → 15.2만 (+27%)
- Revenue: $xxx → $xxx (+18%)
- Churn: 5.2% → 4.1% (-1.1pp)

[PM 판단 필요] 통합 뷰를 구성합니다. 어떤 관점을 강조할까요?
A) 목표 달성률 중심 (낙관적)
B) 미완료 항목과 교훈 중심 (솔직)
C) 지표 성장 트렌드 중심 (데이터 드리븐)
```

**Step 2: 스토리라인 구성**

```
Terminal 입력:
> B와 C를 결합해줘.
  CEO는 "솔직하되 데이터로 뒷받침되는" 스타일을 좋아해.
  10장 슬라이드의 스토리라인을 먼저 보여줘.

Claude Code 응답:
📑 QBR 슬라이드 스토리라인 (10장):

1. TL;DR — 한 줄 요약 + 핵심 숫자 3개
2. OKR 스코어카드 — 3개 OKR × 달성률 시각화
3. 핵심 지표 대시보드 — MAU, Revenue, Churn 트렌드
4. 이번 분기 하이라이트 — Top 3 릴리즈와 임팩트
5. 유저 반응 — NPS 변화 + Slack에서 가장 반응 좋았던 기능
6. 미달성 항목 분석 — 3개 미완료 에픽의 원인과 학습
7. 교훈 & 회고 — "다음 분기에는 이렇게 하겠다"
8. 다음 분기 우선순위 — Top 3 이니셔티브
9. 리소스 요청 — 추가 인원/예산 필요 시
10. Q&A 예상 질문 — CEO가 물어볼 법한 질문 3개와 답변 준비

[PM 판단 필요] 이 구성으로 진행할까요?
슬라이드 6~7이 가장 민감합니다. 미달성의 원인을
"외부 요인"과 "우리 판단 오류"로 정직하게 나눌까요?
```

> **💡 PM 판단 포인트**: QBR의 핵심은 "숫자"가 아니라 "학습"입니다. Claude가 4개 소스에서 수집한 데이터를 단순 나열하는 것이 아니라, "왜 이 결과가 나왔고, 다음에 무엇을 다르게 할 것인지"의 스토리로 엮는 것이 PM의 역할입니다.

## 시나리오 3: 크로스채널 피드백 합성

### 상황

신규 기능을 출시한 지 2주가 지났습니다. 피드백이 여러 채널에 흩어져 있어서 전체 그림을 파악하기 어렵습니다: Slack(내부 피드백), Zendesk(CS 티켓), 인앱 설문(NPS), 앱스토어 리뷰, 트위터 멘션.

### 워크스루

**Step 1: 채널별 피드백 수집**

```
Terminal 입력:
```

```
- 앱스토어: "무료 대안이 있는데 왜 유료?"
- 비즈니스 임팩트: 🟡 중간 — 전환율에 영향, 장기적 성장 제약
- [PM 판단 필요] 가격 조정? 무료 티어 확대? 가치 전달 개선?

### 테마 3: UX 온보딩 혼란 [긴급도: 🟡 중간]
- 범위: 3/4 채널 (Zendesk, NPS, Slack)
...
```

> **💡 PM 판단 포인트**: 크로스채널 합성에서 가장 중요한 것은 "범위(breadth)"입니다. 한 채널에서만 나오는 이슈는 채널 특성일 수 있지만, 4개 채널 모두에서 나오는 이슈는 진짜 문제입니다. Claude가 범위를 표시해주므로, PM은 "어디서 얼마나 많이"를 기준으로 우선순위를 결정합니다.

## 시나리오 4: 파트너/인수 실사 보고서

### 상황

사업개발팀에서 AI TTS(Text-to-Speech) 스타트업 인수를 검토 중입니다. CEO가 "기술력, 시장 위치, 재무 건전성을 종합적으로 평가한 실사 보고서를 만들어달라"고 요청했습니다.

### 워크스루

**Step 1: 공개 데이터 기반 실사**

```
Terminal 입력:
> "VoiceAI Labs"라는 AI TTS 스타트업을 실사해줘.
  다음 5가지 축으로 분석:

  1. 기술력 평가
     - 공개된 논문, 특허, 기술 블로그
     - GitHub 활동 (오픈소스 기여)
     - 기술 벤치마크 (MOS 점수 등)

  2. 시장 위치
     - 경쟁 구도 (ElevenLabs, Play.ht, Murf 등)
     - 차별화 포인트
     - 고객 사례/리뷰

  3. 팀 역량
     - 창업팀 배경 (LinkedIn 공개 정보)
     - 핵심 엔지니어 수 추정
     - 채용 공고 분석 (성장 방향 추론)

  4. 재무 추정
     - 공개된 투자 라운드
     - 추정 ARR (채용 규모, 고객 수 기반)
     - 번레이트 추정

  5. 리스크 요인
     - 기술 부채, 규제 리스크, 핵심인력 의존도

  각 항목에서 [확인된 사실]과 [추정]을 반드시 구분해줘.

📋 VoiceAI Labs 실사 보고서
```

## 시나리오 5: 멀티소스 데이터 결합 분석

### 상황

분기 실적 보고서를 준비 중인데, 내부 대시보드(Amplitude)의 유저 행동 데이터, 재무팀의 매출 스프레드시트, 그리고 마케팅팀의 캠페인 결과 시트를 하나로 결합하여 "어떤 마케팅 캠페인이 실제 매출로 연결됐는지"를 분석해야 합니다.

### 워크스루

**Step 1: 데이터 소스 결합**

```
Terminal 입력:
> 세 가지 데이터 소스를 결합해서 분석해줘:

  1. @./data/amplitude-export.csv
     — 유저별 활성화, 리텐션, 핵심 이벤트
  2. @./data/revenue-q4.xlsx
     — 월별 매출, 고객별 MRR, 코호트
  3. @./data/marketing-campaigns.csv
     — 캠페인별 지출, 노출, 클릭, 가입

  분석 목표:
  "어떤 마케팅 캠페인 → 어떤 유저 세그먼트 유입 →
   실제 유료 전환 및 매출 기여까지"의 풀 퍼널을 연결해줘.

  조인 키는:
  - amplitude: user_id
  - revenue: customer_id (user_id와 매핑 테이블 있음)
  - marketing: utm_source → amplitude의 initial_utm

Claude Code 응답:
🔗 3개 데이터 소스를 결합합니다...

### 데이터 결합 결과
- Amplitude: 12,340 유저
- Revenue: 890 유료 고객
- Marketing: 8개 캠페인

조인 성공률:
- Amplitude ↔ Revenue: 92% 매칭 (user_id ↔ customer_id)
- Marketing ↔ Amplitude: 78% 매칭 (utm 기반)

⚠️ [PM 판단 필요] 22%의 유저는 UTM이 없어 캠페인 귀속 불가.
이들을 "Organic"으로 분류할까요, 아니면 분석에서 제외할까요?
```

```
Terminal 입력:

  캠페인 → 가입 → 활성화(Day 7) → 유료 전환 → LTV 기여

Claude Code 응답:
📊 캠페인별 풀 퍼널 분석:

| 캠페인 | 지출 | 가입 | Day7 활성 | 유료 전환 | MRR 기여 | CAC | LTV/CAC |
|--------|------|------|-----------|----------|---------|-----|---------|
| Google SEM | $12K | 2,100 | 45% | 8.2% | $4,200 | $70 | 2.1x |

| Product Hunt | $0 | 1,200 | 71% | 12.3% | $2,900 | $0 | ∞ |
| YouTube Sponsor | $5K | 890 | 38% | 4.1% | $890 | $137 | 0.9x |
| Organic | - | 3,400 | 58% | 11.7% | $8,100 | $0 | ∞ |
| ... | | | | | | | |

### 핵심 인사이트:
1. **Product Hunt**이 CAC $0에 가장 높은 활성화율 (71%)
```

## 시나리오 6: 대시보드 지표 자동 추출

### 상황

매주 월요일 아침, 경영진 위클리에 쓸 핵심 지표를 Amplitude, Google Analytics, Stripe Dashboard에서 하나씩 복사-붙여넣기하고 있습니다. 매주 30분씩 반복됩니다.

### 워크스루

**Step 1: Claude in Chrome으로 대시보드 순회**

```
Terminal 입력 (Claude in Chrome):

  2. Google Analytics → 세션 수, 이탈률, 전환율
  3. Stripe Dashboard → MRR, 신규 구독, Churn

  각 지표를 전주 대비 변화율과 함께 정리하고,
  이상 징후(전주 대비 ±10% 이상)가 있으면 하이라이트해줘.

Claude in Chrome 응답:
📊 주간 지표 추출 완료 (2026-02-16 ~ 02-22):

|---------|------|---------|------|--------|------|
| Product | WAU | 15,230 | 14,890 | +2.3% | ✅ 정상 |
| Product | D7 Retention | 42.1% | 43.8% | -1.7pp | ✅ 정상 |
| Product | D30 Retention | 28.5% | 27.2% | +1.3pp | ✅ 정상 |

| Traffic | Conversion | 3.8% | 4.2% | -0.4pp | ✅ 정상 |
| Revenue | MRR | $48.2K | $46.8K | +3.0% | ✅ 정상 |
| Revenue | New Subs | 67 | 58 | +15.5% | ⚠️ 관찰 필요 |
| Revenue | Churn | 2.1% | 1.8% | +0.3pp | ⚠️ 주의 |

### 🚨 이상 징후 분석:
1. **세션 +10% but 이탈률 +4.2pp**

   → 마케팅 캠페인 변경? 봇 트래픽? 확인 필요

2. **신규 구독 +15.5% but Churn +0.3pp**
   → 유입은 증가했으나 기존 고객 이탈도 증가
   → [PM 판단 필요] 기능 변경이 기존 유저에 영향?
```

## 핵심 요약

이 시나리오들의 공통 패턴:

1. **분산 데이터 → 통합 뷰**: PM의 가장 큰 고통은 "정보가 여기저기 흩어져 있다"는 것. Claude는 MCP/파일/웹을 넘나들며 수집합니다.

2. **[사실] vs [추정] 구분**: Claude가 모든 정보를 사실처럼 제시하면 위험합니다. 좋은 프롬프트는 이 구분을 명시적으로 요구합니다.

3. **PM 판단 지점 표시**: Claude가 "여기서 당신이 결정해야 합니다"라고 멈추는 것이 가장 중요한 기능입니다. 자동화의 목적은 "PM의 판단을 대체"하는 것이 아니라 "PM이 판단에 집중할 수 있게"하는 것입니다.

4. **반복 → 자동화**: 시나리오 6처럼 매주 반복되는 작업은 스케줄 태스크로 전환하여 PM의 시간을 전략적 사고에 재배분합니다.
